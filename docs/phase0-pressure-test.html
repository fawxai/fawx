Canary → Ripcord Wiring
~150 lines
🔴 Scope underestimated — needs rethink
Findings
Risk Canary is ALSO unwired. fx-canary is pure computation — Canary::evaluate() takes a &[Signal] and returns a Verdict. But nothing ever calls it. No background task, no periodic check. The canary doesn't run anywhere — it's library code like the fleet.
Risk No signal collection mechanism. Canary needs signals to evaluate. Who collects them? The loop engine emits signals, but there's no signal aggregator that accumulates them for canary evaluation. We'd need to build: signal collector → periodic canary evaluation → ripcord trigger.
Risk Ripcord snapshots are manual only. fawx-ripcord --create creates snapshots manually. There's no auto-snapshot on deploy/skill-install. For canary→ripcord to work, we need automatic pre-deployment snapshots.
Issue Ripcord restores config.toml + skills. But what about the fawx binary itself? It hashes it but doesn't restore it. If a bad binary was deployed, ripcord can only restore config/skills.
Key Decisions
Decision Revised scope: This PR is actually 3 pieces: (a) signal collector in the loop engine, (b) periodic canary evaluation as background task, (c) ripcord trigger on Degraded verdict. Estimate jumps from ~150 to ~350 lines.
Decision Auto-snapshot timing: When should snapshots be created automatically? Options: (1) before every skill install/update, (2) on a daily schedule, (3) both. Recommend: before skill install + daily cron.

4
Proposal Approval Flow
~300 lines
⚠ Design correction needed
Findings
Risk Spec over-engineers the approval path. /approve is a HUMAN action, not an agent action. It doesn't need to go through ProposalGateState.set_active_proposal() — that mechanism is for granting the agent temporary write permission. The human can just write the file directly. Simpler: read proposal → write file → move to applied/.
Risk Stale proposal danger. If the target file changed between proposal creation and approval, applying the proposal overwrites the changes. Proposals store full file content, not diffs. Need a staleness check: compare file hash at proposal time vs current.
Issue Proposal format parsing is fragile. ProposalWriter::format_proposal() creates markdown. Parsing it back requires exact format matching. If the writer format ever changes, the parser breaks. Consider: write a structured sidecar (JSON) alongside the markdown.
OK Tier 3 check on approve is correct and necessary — even humans shouldn't casually modify kernel paths through this flow.
Key Decisions
Decision Drop ProposalGateState involvement. /approve writes directly as the human. No need for set_active_proposal(). This removes the refactor of ProposalGateExecutor to accept Arc<Mutex<ProposalGateState>> — significant simplification.
Decision Staleness guard: ProposalWriter should save target file hash in proposal metadata. /approve checks: if file hash changed since proposal → warn and require /approve <id> --force.
Decision Structured sidecar: Write proposal-{id}.json alongside proposal-{id}.md. JSON has: target_path, proposed_content, file_hash_at_creation, risk, timestamp. Markdown is human-readable, JSON is machine-parseable.

ChannelRegistry + ResponseRouter Wiring
~400 lines
✓ Architecture verified — solid
Findings
OK Channel trait is well-designed. TelegramChannel implements Channel. ChannelRegistry supports register/get/list/remove. ResponseRouter routes by InputSource. All tested. The abstraction is ready.
OK ChannelRegistry.register() takes Box<dyn Channel> — but the spec says Arc. Needs to be Box. Minor spec fix.
Issue Telegram long-polling is pull-based. Current run_telegram_polling() is a background task that calls telegram.get_updates() in a loop. This doesn't fit cleanly into ChannelRegistry's push model. The polling task needs to continue as-is, but route received messages through the registry.
Issue Response routing is async challenge. Agent processing is async (send message → loop runs → response ready). HttpChannel uses a Mutex<Option<String>> slot. Telegram needs to send responses back via send_message API. The Channel::send_response() trait is sync (&self, returns Result). Telegram's send is async. May need send_response to accept a tokio channel sender instead.
Key Decisions
Decision Async response delivery: TelegramChannel::send_response() can't directly await the Telegram API. Two options: (1) buffer response internally and have polling task drain it, or (2) change trait to async. Recommend option 1 — same pattern as HttpChannel's pending_response slot. Keep the trait sync.
Decision Telegram polling stays as background task — receives messages, tags with InputSource::Channel("telegram"), feeds into the loop engine. ResponseRouter sends responses back via the channel's buffered slot. Polling task checks the buffer on each iteration.
6
fx-fleet Wiring
~500 lines
🔴 Premature — defer or radically rescope
Findings
Risk Fleet assumes Fawx nodes, but we have SSH machines. The fleet model (heartbeat, capability tags, stale detection) assumes fleet nodes that actively report in. Our actual nodes are Mac Mini and MacBook that we SSH into. They don't run a Fawx fleet agent. The heartbeat model doesn't match reality.
Risk Task dispatch overlaps with node_run. PR 3 wires node_run for direct SSH command execution. The fleet orchestrator also dispatches tasks to nodes. These are competing mechanisms for the same thing — risk of confusion about which path to use.
Risk No fleet client exists. Even if we wire the server-side orchestrator, there's no fleet client running on remote nodes to receive tasks, execute them, and report results. The orchestrator can submit and track tasks but has nothing on the other end.
Issue Highest risk PR in Phase 0. 500 lines touching HTTP message flow, background services, and distributed coordination — with no client-side implementation and a model that doesn't match our current infrastructure.
Key Decisions
Decision Recommend: defer to Wave 9 or later. Fleet orchestration is future infrastructure for when Fawx runs on multiple nodes as a distributed system. Today, node_run via SSH (PR 3) covers our actual use case. Don't wire a distributed orchestrator to a system that only has SSH nodes.
Decision If we keep it: Radically rescope to just NodeRegistry from config (read-only, no heartbeat) + fleet status endpoint (report config, not live state). Drop TaskRouter and Orchestrator wiring entirely. ~100 lines instead of 500.
