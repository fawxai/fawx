Kernel + Loadable Layers + Platform Shell — designed for self-reconfiguration with strict safety

Pre-Dream Checkpoint Protocol

Memory consolidation ("dreaming") is the most dangerous self-modification the system performs — it rewrites persistent memories. Unlike loadable layer changes (protected by A/B slots), bad consolidation can destroy memories the user doesn't even realize are gone until weeks later. The checkpoint protocol makes this recoverable.

🔒 Consolidation cannot begin without a validated checkpoint. The dreaming process cannot start until an atomic snapshot of all memory stores has been created, written, and integrity-verified. No API exists to bypass this gate.

What Gets Checkpointed

The snapshot is atomic across all persistent memory stores. Partial snapshots are never valid.

Store Snapshot includes Why
Episodic Memory All entries, timestamps, importance scores, decay state Consolidation decays and prunes entries — must be restorable
Semantic Memory All facts, embeddings, indexes, confidence scores Consolidation promotes episodic→semantic — bad promotions must be reversible
Procedural Memory Strategy versions, A/B slot state, performance metrics Dream cycle may update strategy scores — rollback restores prior evaluations
Identity Profile All dimensions, confidence scores, signal history Dreaming consolidates identity signals — bad consolidation can distort the profile

Not checkpointed: Working memory (ephemeral by design — dies with the task), audit log (append-only, never modified by consolidation).

Checkpoint Retention Policy

The user controls retention. The kernel enforces it.

Retention Controls (User-Only)

Recent checkpoints to keep default: 7
User-pinned checkpoints never pruned
Emergency checkpoint (last pre-anomalous-dream) always retained
Storage budget for checkpoints user-configurable
Oldest unpinned exceeding budget auto-pruned

Post-Dream Health Check

After consolidation, the kernel validates memory integrity before releasing the checkpoint safety net:

• Entry count sanity — did consolidation delete an anomalous percentage of entries? (Threshold: user-configurable, default 30%)
• Index consistency — do all index entries point to valid memory entries? No orphaned references.
• Semantic coherence — do newly promoted semantic facts have valid provenance chains to source episodic memories?
• Identity stability — did any identity dimension confidence score change by more than the anomaly threshold in a single cycle?
• Storage integrity — checksums on all encrypted stores match expected values.

Design principle: A/B slots protect the loadable layer (config, skills, strategies). The checkpoint protocol protects the storage layer (memories, identity, learned knowledge). Together they guarantee: any self-modification the system performs is recoverable. The system never bricks. The agent never forgets — unless the user tells it to.

Kernel Invariants

These properties are guaranteed by compiled Rust in the kernel. They cannot be overridden by skills, config, or the agent.

🔒 Permission registry is user-only. The agent cannot escalate its own permissions. No skill, config change, or prompt injection can modify what the agent is allowed to do.
🔒 Policies are user-only by default. Action policies (allow/deny/confirm rules) can only be changed by the user. The agent operates within the policy, never modifies it.
🔒 Rollback cannot be disabled. The A/B slot mechanism and watchdog timer are always active. No API exists to turn them off.
🔒 Audit trail is append-only. Every modification attempt (approved or denied) is logged. Neither the agent nor skills can edit or delete log entries.
🔒 Capability dropping is one-way. A skill granted fewer capabilities than it requests can never gain more. Escalation is structurally impossible.
🔒 The kernel is immutable at runtime. The loop orchestrator, state machine, policy engine, and all enforcement mechanisms are compiled Rust. They cannot be modified, patched, or replaced while the system is running.
🔒 Consolidation cannot proceed without a validated checkpoint. Before any memory consolidation ("dreaming") cycle, the kernel creates and verifies an atomic snapshot of all persistent memory stores. If the snapshot fails integrity checks, consolidation is blocked. If post-consolidation health checks fail, the system rolls back to the checkpoint automatically. No API exists to bypass this gate.
Design principle: The kernel is the smallest possible set of guarantees that cannot be violated. Everything else — intelligence, capabilities, strategies, UI — lives above the kernel in layers the agent can evolve. Safety comes from architecture, not from trust.

Extensibility: Agent Identity Primitives

Emerging patterns — agent-owned email addresses (agentmail), on-chain wallets, and machine-to-machine payment protocols (x402) — are gaining traction. Fawx does not implement these today, but the four-layer architecture accommodates them without kernel changes:

Agent email New AuthMethod::Email variant in the storage layer. Sending/receiving maps to existing perception (input) and action (output) loop steps. Wallets New AuthMethod::Wallet { chain, address } credential. Signing keys stored encrypted alongside API keys. Transaction authorization governed by existing budget/policy enforcement in the kernel — user sets a spending envelope, agent operates within it. Payment protocols New loadable tools (pay_402, sign_transaction). Hot-swappable, no kernel recompile. Budget system extends from token limits to monetary limits via a data-only change. Agent-to-agent comms Messages from other agents are just another perception source. The loop doesn't care whether input comes from a human, an email, or another agent — it's all PerceptionSnapshot.

The critical invariant holds: user-only permissions. An agent with a wallet still cannot authorize spending beyond what the user's policy allows. The kernel enforces the envelope; the loadable layer provides the capabilities.
