Memory-Augmented Intelligence

Four ascending layers of intelligence improvement — from context injection to genuine learning. Each layer makes the agent measurably better at serving its human. Each is achievable with increasing capability requirements.

The question: Can the agent's intelligence actually improve from experience, not just its knowledge? The answer is yes — through four distinct mechanisms, each more powerful than the last. The architecture supports all four from day one, even though the higher layers won't be active until later.

The Four Intelligence Layers

Layer 1 — Retrieval-Augmented Generation (RAG) BUILD NOW

PHASE 1

Mechanism: At the Perceive step, relevant memories are retrieved and injected into the LLM prompt. The model's weights don't change — but its effective behavior changes based on what context it receives.

How it improves over time: Importance-weighted retrieval (from the dreaming cycle) means frequently-useful memories naturally rise to the top of retrieval results. The agent gets better at surfacing the right context because the memory system learns which memories matter.

Layer 2 — Behavioral Conditioning via Procedural Memory BUILD NOW

PHASE 1

Mechanism: Procedural memory (strategies, prompt templates, recovery patterns) directly conditions the agent's behavior. When the agent learns that "for navigation tasks, element IDs are more reliable than text matching," this is encoded as a strategy that modifies the system prompt. The LLM's weights don't change, but its instructions do — based on experience.

How it improves over time: Strategy A/B testing. When procedural memory proposes a new strategy (via A/B slots), the kernel can run both strategies for N tasks and compare success rates. The strategy with higher verified-success rate wins. This is evolution — not gradient descent, but it produces measurable behavioral improvement.

Strategy Evolution Cycle

1. Agent encounters repeated pattern (e.g., text input fails 40% of the time)
2. During Learn step, pattern is recorded in episodic memory
3. During dreaming, pattern detection identifies the cluster
4. System proposes strategy update: "use element ID selectors instead of text matching"
5. New strategy → pending A/B slot
6. Kernel runs both strategies for next N tasks of matching type
7. Compare verified success rates
8. Winner promoted to active slot; loser archived
ComponentLayerDetails
Strategy proposalLoadableDreaming consolidation detects patterns → proposes strategy. Requires LLM inference for strategy formulation.

Comparison metricsKernelVerification success rate, retry rate, average step count. Kernel tracks per-strategy metrics.
Promotion decisionKernelStatistical significance threshold before promoting. Minimum N tasks before evaluation.

Measurement: Strategy win rate. Track how often newly evolved strategies outperform their predecessors. This is the closest thing to a "learning rate" for behavioral improvement.

Layer 3 — Learned Retrieval Attention DESIGN NOW, BUILD LATER

PHASE 2-3

Mechanism: Instead of fixed re-ranking functions, a small neural network learns to predict which memories will be useful for a given context. The network maps (context_embedding, memory_embedding) → relevance_score, trained on whether retrieved memories contributed to successful outcomes.

How it improves over time: The attention network is trained continuously on retrieval outcome data. Every time a retrieved memory contributes to a verified-successful task, that's a positive training signal. Every time a high-ranked memory is unused, that's a negative signal. The network learns what to remember when.

Architecture

// Retrieval with learned attention:
fn retrieve(context: &Context, store: &MemoryStore) -> Vec<MemoryHit> {
    let ctx_emb = embed(context);
    let candidates = store.top_k_similar(ctx_emb, k=100);

    // Layer 1-2: fixed re-ranking
    // Layer 3: learned attention network
    let scored = attention_net.score_batch(ctx_emb, candidates);
    scored.sort_by_relevance().take(budget)
}

// Training signal (collected during Learn step):
// For each retrieved memory in this cycle:
//   if task_verified_success AND memory was in context: positive
//   if task_verified_success AND memory NOT in context: neutral
//   if retrieved but task failed: weak negative
//   if retrieved but never referenced by LLM: negative
RequirementDetails
ModelSmall (< 10M params) neural network. Two-tower architecture: context encoder + memory encoder → dot product.
TrainingOn-device, incremental. Runs during dreaming cycles (after Phase 4/Prune, before Phase 6/Audit). Small batch, few epochs.
Training dataRetrieval outcome log — automatically collected during normal operation. No labeling needed.
FallbackIf the attention network performs worse than the fixed re-ranking function (measured over N cycles), automatically fall back to Layer 2. A/B slot pattern.
PrivacyEntirely on-device. No training data or model weights leave the device.

Key insight: This is a genuine Memory-Augmented Neural Network (MANN). The system learns what to remember and when to remember it, not just what it knows. The difference between Layer 2 (heuristic re-ranking) and Layer 3 (learned attention) is the difference between rules-of-thumb and intuition.

Measurement: Retrieval precision@K (did the top K results actually get used?), NDCG (was the ranking order optimal?), downstream task success rate improvement vs Layer 2 baseline.

Layer 4 — Weight Modification (On-Device Fine-Tuning) RESEARCH PHASE

OS ERA — DESKTOP FIRST

Mechanism: True model personalization — fine-tuning the local LLM on the user's interaction history, memory store, and preference data. The model literally gets smarter about this specific user over time.

How it improves over time: The model internalizes patterns that can't be captured by prompt injection alone. Vocabulary, reasoning patterns, domain knowledge, communication style — baked into the weights. The effective context window becomes infinite because the model has learned the user's context, not just been told about it.

Safety Considerations

Fine-tuning can degrade model quality. Unlike Layers 1-3 (which add intelligence without modifying the model), Layer 4 modifies the model itself. Bad training data, overfitting, or catastrophic forgetting could make the model worse.

Mitigation:

• Keep the base model intact. Fine-tuning produces a LoRA adapter, not a modified base model. Rollback = remove the adapter.
• A/B evaluation before promoting. Run the fine-tuned model and base model side by side for N tasks, compare verified success rates.
• Pre-fine-tuning model checkpoint (analogous to pre-dream memory checkpoint). If the fine-tuned model underperforms, automatic rollback.
• User controls: "fine-tune my model" is an explicit opt-in, not a default behavior.

Day-One Design Decisions

Even though Layers 3 and 4 won't be active in Phase 1, the architecture must support them from the start. These decisions cannot be retrofitted easily.

DecisionWhy nowCost of deferring
Every memory gets an embedding vector Layer 3 needs embeddings to train the attention network. Layer 1 needs them for retrieval. Building without embeddings means re-indexing the entire memory store later. Full memory re-indexing migration. Potential downtime during transition.
Every retrieval is logged with outcome data Layer 3 training signal comes from retrieval outcomes. Layer 2 strategy evolution needs this data. Without it, there's no training data when Layer 3 activates. Months of lost training data. Layer 3 starts cold instead of warm.
Retrieval interface is abstract The retrieval strategy must evolve from keyword → embedding → learned attention → fine-tuned without changing the kernel. The kernel calls retrieve(context) → MemoryHit[]; everything else is strategy. Kernel changes to support new retrieval modes. Risky, expensive, and breaks the immutability invariant.
Embedding model is configurable On-device embedding models will improve rapidly. The system must be able to swap models without re-architecting. Hardcoded embedding dimensions. Model upgrade requires full re-embed of all memories.
Memory importance scoring is loadable Scoring weights will evolve. Layer 2 tunes them via A/B slots. Layer 3 replaces them entirely. The kernel must not hardcode a scoring function. Kernel change to support learned scoring. Undermines the kernel/loadable boundary.

Measuring Intelligence Improvement

Each layer has measurable signals that indicate whether the agent is getting smarter.

LayerPrimary metricSecondary metrics
Layer 1 (RAG) Memory contribution rate: % of successful tasks where retrieved memories were used in reasoning Retrieval precision@K, context utilization ratio, duplicate retrieval rate
Layer 2 (Behavioral) Strategy evolution win rate: % of new strategies that outperform predecessors Task success rate trend, retry rate trend, average actions per task trend
Layer 3 (Learned attention) Retrieval NDCG improvement vs Layer 2 baseline Retrieval precision@K, downstream task success delta, training loss convergence
Layer 4 (Fine-tuning) Task success rate: fine-tuned vs base model (controlled comparison) First-attempt success rate, style match score, user correction frequency
The compound effect: Layers stack. Layer 1 makes retrieval useful. Layer 2 makes strategies evolve. Layer 3 makes retrieval learn. Layer 4 makes the model itself learn. Each layer amplifies the ones below it. An agent running all four layers isn't 4x smarter — it's compounding intelligence improvements across every dimension of the system.
