Anticipation Loop

Proactive background perception — the agent watches for conditions and offers help before the user asks. Transforms the agent from reactive tool to anticipatory partner.

The difference: A reactive agent waits for "book me a flight." An anticipatory agent notices you have a meeting in Denver tomorrow, you haven't booked a flight, and it's getting late — and offers to help. This is the line between a tool and a partner. But it must be done carefully: a proactive agent that guesses wrong is annoying. A proactive agent that acts without permission is dangerous.

Architecture

The Anticipation Loop is a background subsystem that runs during idle time. It uses a lightweight version of the main loop's Perceive step, feeds observations through a trigger classifier, and produces suggestions — never autonomous actions.

Kernel Module: Anticipator

Anticipator {
  // Kernel-owned state
  timer:              IdleTimer,        // Fires perception checks during idle
  suppression_state:  SuppressionState, // Tracks fatigue / dismiss patterns
  proactive_policy:   ProactivePolicy,  // Stricter than user-initiated policy

  // Lifecycle
  fn on_idle_tick():
    if user_active() || task_running(): return
    if suppressed(): return

    perception = lightweight_perceive()  // sensors + notifications + time + calendar
    triggers = classifier.evaluate(perception, identity_profile)

    for trigger in triggers:
      if trigger.confidence >= threshold:
        suggestion = formulate_suggestion(trigger)
        emit_proactive_suggestion(suggestion)
        update_fatigue_counter()
}
ComponentLayerDetails
Timer management Kernel Idle detection, tick scheduling, integration with OS doze/battery optimization. Respects system power state.
Lightweight perceive Platform Kernel Reads: current time, calendar events, notification queue, battery/network state, location (if permitted). Does NOT read full screen content (too expensive for background checks).
Trigger classifier Loadable Evaluates perception snapshot against known trigger patterns. Returns trigger + confidence score. Evolves via A/B slots as the agent learns what the user finds useful vs annoying.
Suggestion formulation Loadable Composes the proactive message. Uses identity profile for tone, context for specificity. May require lightweight LLM call (or can be template-based for low-power scenarios).
Proactive policy enforcement Kernel All proactive suggestions pass through the Decide gates with elevated policy requirements. Default: all proactive actions require confirmation. User can relax specific categories.
Fatigue management Kernel Tracks dismiss rate. If user dismisses ≥3 consecutive suggestions, suppression activates (doubles the idle interval). Resets on user acceptance.

Trigger Tiers

Not all proactive triggers are created equal. They're organized in three tiers by urgency and confidence requirements.

Tier 1 — Time-Critical Alerts

Confidence threshold: 0.9+ (very high — these interrupt)

Delivery: Immediate notification, even during light idle.

Examples:

• Meeting in 15 minutes, haven't left home yet, commute is 20 minutes
• Flight in 3 hours, haven't checked in, airline allows online check-in
• Unusual charge on credit card notification just arrived
• Birthday of close contact today, haven't sent a message yet

Signal sources: Calendar + location + time + notification content + social graph

Tier 2 — Contextual Suggestions

Confidence threshold: 0.75+ (high — these suggest, don't interrupt)

Delivery: Ambient UI element (badge, subtle card). Shown during natural interaction pauses.

Examples:

• At a restaurant the agent hasn't seen before → "Want me to save this to your places?"
• Weekly grocery order usually placed on Sundays, it's Sunday afternoon → "Want me to start your grocery order?"
• Approaching end of billing cycle, usage is high → "You've used 80% of your data plan"
• Repeated searches for a topic → "I found some related articles you might like"

Signal sources: Location + routines + episodic patterns + identity preferences

Tier 3 — Passive Insights

Confidence threshold: 0.6+ (moderate — these are offered, never pushed)

Delivery: Available on request or during daily summary. Never interrupts.

Examples:

• "You've been ordering from Thai Palace a lot — they have a loyalty program"
• "Your meeting pattern this week is heavier than usual — want me to block some focus time?"
• "You haven't backed up your photos in 2 weeks"
• "Based on your sleep patterns, you seem to get better rest when you stop using your phone by 11pm"

Signal sources: Episodic patterns + semantic knowledge + routine analysis + dreaming consolidation insights

Power Management

The anticipation loop runs on a mobile device. Power consumption must be carefully managed.

Power stateAnticipation behaviorTick interval
Charging + idle Full anticipation. All tiers active. LLM-based trigger classification if needed. 5 minutes
Battery > 50% + idle Standard anticipation. Tier 1 and 2 active. Template-based classification (no LLM). 15 minutes
Battery 20-50% + idle Reduced anticipation. Tier 1 only. Minimal sensor reads. 30 minutes
Battery < 20% Anticipation suspended. Only OS-level notifications processed. Suspended
Active use Anticipation paused. Main loop has priority. Paused

Integration with platform scheduling: The Anticipator uses the shell's background execution mechanism — cron/timers on desktop/TUI, Android's WorkManager on mobile, iOS's BGTaskScheduler on Apple devices. It respects platform power constraints. The user can always disable background activity entirely.

Fatigue Prevention

A proactive agent that suggests too often is worse than a reactive one. The kernel manages suggestion fatigue with multiple mechanisms.

MechanismBehaviorReset condition
Dismiss tracking 3 consecutive dismissals → double the tick interval for this tier User accepts a suggestion from the suppressed tier
Daily cap Maximum suggestions per day per tier: Tier 1 = 5, Tier 2 = 3, Tier 3 = 2 Resets at midnight local time
Context dedup Don't suggest the same thing twice within 24 hours unless the trigger strengthens 24-hour window per trigger type
User mood signal If the user's recent interactions have been curt/dismissive, suppress Tier 2-3 Interaction pattern normalizes
Explicit suppression "Stop suggesting X" → category-level suppression User re-enables or time-based expiry (user-configurable)

Learning from Feedback

Every proactive suggestion generates a learning signal, whether accepted, dismissed, or ignored.

ProactiveFeedback {
  trigger:     TriggerType,
  tier:        Tier,
  confidence:  f32,
  outcome:     Accepted | Dismissed | Ignored | ActedOnLater,
  context:     PerceptionSnapshot,  // What was happening when suggestion was made
  timing:      Instant,
}

// Learning signals:
// Accepted          → strong positive (this trigger is useful)
// Dismissed         → moderate negative (wrong time or wrong suggestion)
// Ignored           → weak negative (might have been useful but not enough)
// ActedOnLater      → positive with timing correction (right suggestion, wrong time)
//
// These signals feed into:
// 1. Trigger classifier evolution (A/B slots for classifier params)
// 2. Timing model refinement (when to suggest)
// 3. Fatigue threshold tuning
// 4. Episodic memory (for dreaming consolidation of proactive patterns)

Over time: The classifier learns which triggers the user finds useful vs annoying. Timing shifts toward when the user is most receptive. Confidence thresholds self-calibrate. The agent becomes better at offering the right thing at the right time — and learns to stay quiet when it should.

Anticipation Invariants

🔒 Proactive mode is suggest-only by default. The anticipation loop cannot execute actions autonomously. All proactive outputs are suggestions that require user confirmation. The user can relax this per-category, but the kernel default is always confirm-required.
🔒 Proactive policy is strictly tighter than reactive policy. Any action that requires confirmation in reactive mode also requires confirmation in proactive mode. Proactive mode cannot have more permissive policies than the user has set for reactive interactions.
🔒 The user can disable anticipation entirely. A single toggle in settings disables the entire Anticipation Loop. The kernel respects this unconditionally.
🔒 Anticipation never runs during active tasks. The main loop has absolute priority. Anticipation pauses the moment any user interaction begins.
🔒 Fatigue management is kernel-enforced. The daily cap and dismiss-based suppression cannot be overridden by loadable strategies. The classifier can get better at suggestions, but it can't override the rate limits.
Design principle: Proactive intelligence is the most powerful and most dangerous capability of a personal agent. Powerful because it transforms the relationship from command-response to partnership. Dangerous because unsolicited actions erode trust fast. The architecture ensures proactive behavior is always gated by high confidence, always suggestion-only by default, always fatigue-managed, and always disableable. The agent earns the right to anticipate by being right — and learns to stay quiet when it's not.
