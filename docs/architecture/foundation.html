Foundation Decisions

Seven structural decisions that define Fawx's architecture. Ratified and implemented. The engine is built on these.

7 DECISIONS — ALL RATIFIED & IMPLEMENTED
DECISION 1

Four-Layer Separation

The system is four layers with strict boundaries. Higher layers can be modified at runtime; the kernel cannot.

Shell Layer
TUI (fx-cli), future: Android (Kotlin), iOS (Swift), Desktop (Tauri). Renders UISpec, captures input, executes platform actions.

DECISION 2

Shells Drive, Engine Thinks

The boundary between shell and engine is pure request-response. The shell provides input (user messages, perception data), the engine runs the loop and returns decisions. No callbacks from engine into shell. The engine is a stateless-looking function: data in, decisions out.

Shell
Captures user input
Renders UISpec
Executes actions
Reports results
input → →
Rust Engine
Runs loop
Makes decisions
Manages memory
Enforces policy
← decisions + UISpec ←
Shell
Executes decisions
Returns results
Repeats

TUI shell (current): fx-cli calls engine directly as a Rust library. No FFI needed — it's the same process.

Future native shells: UniFFI (Mozilla) generates Kotlin + Swift bindings from Rust. The generated code is readable and forkable.

Shell Boundary: One Fat Call

The shell sends a full input snapshot, gets back all decisions for the cycle. The engine runs the full loop internally. The shell's job is: gather input, send it in, get decisions back, execute them. Iterate.

Why this wins: Cleaner boundary. Engine is self-contained. Adding a new shell (iOS, desktop, web) is trivial — same API, different renderer. The Rust side owns the entire loop; the shell is just I/O.

DECISION 3

Seven-Step Recursive Loop

Every agent action follows the same cycle. The loop is a recursive function: Loop(goal, context, depth) → LoopResult. Sub-goals, evidence gathering, and recovery all spawn child loops with managed depth, scoped context, and independent checkpoints.

1 Perceive
→
2 Reason
→
3 Decide
→
4 Act
→
5 Verify
→
6 Learn
→
7 Continue
Recursion & Re-entry Points
3 → 2 Denied — intent rejected by policy/budget/permission. Re-enter Reason with denial as constraint. 3 → ↻ Decompose — goal too complex. Spawn child Loop(sub_goal, depth+1). 5 → 4 Retry — verification failed, retry same action. 5 → 2 Replan — try different approach. Back to Reason with failure context. 7 → 1 Continue — more work needed. Fresh perception cycle.
↻ = spawns child loop. Max depth: 3 (kernel-enforced). Child inherits parent context (read-only), gets scoped budget, checkpoints independently.
DECISION 4

Four Memory Types

Memory is four distinct systems. This determines the storage crate's module structure.

Working
Conversation context, active task state. The scratchpad.
Ephemeral — dies with the task. Kernel-managed.
Episodic
What happened, when, what the outcome was. Interaction history.
Persistent, time-decaying. Storage layer.
Semantic
Durable facts, preferences, identity. Things that are true.
Persistent, curated. Storage layer.
Procedural
Strategies, prompt templates, recovery patterns. How to do things.
Persistent, versionable. Loadable layer (A/B slotted).
DECISION 5

Kernel Invariants

These are the guarantees the kernel enforces in compiled Rust. They cannot be overridden by skills, config, or the agent.

🔒 Permission registry is user-only. The agent cannot escalate its own permissions.
🔒 Policies are user-only. Action allow/deny/confirm rules can only be changed by the user.
🔒 Rollback cannot be disabled. A/B slots and watchdog are always active.
🔒 Audit trail is append-only. Neither the agent nor skills can edit or delete log entries.
🔒 Capability dropping is one-way. Skills cannot gain capabilities beyond what they were granted.
🔒 Kernel is immutable at runtime. Loop orchestration, state machine, policy engine — compiled Rust, never patchable.
🔒 Consolidation requires checkpoint. Memory dreaming cannot start without a validated atomic snapshot.
🔒 Three-gate decision. Every action passes policy → budget → permission. No gate can be skipped.
DECISION 6

TUI-First: Validate the Loop Before Building GUI

The terminal interface is the first shell. It validates that the agentic loop, auth, tool execution, and memory work correctly before investing in native GUI shells.

Why TUI first:

• Fastest iteration cycle — no build/deploy/install loop
• The engine's value is in the loop, not the UI. A terminal proves the loop works.
• Self-bootstrapping: once tool execution works, Fawx implements its own features. The agent reads its own code, writes changes, runs tests. "Fawx builds Fawx."
• GUI shells (Android, iOS, desktop) come after the engine is proven. They render UISpec — a declarative contract the engine already defines.

What this means for the codebase: fx-cli is a first-class shell, not a debug tool. It has its own auth wizard, conversation management, command system, and eval harness. It's 5,449 lines of production code.

DECISION 7

Subscription Auth as First-Class Citizen

Most users already pay for an AI subscription. Fawx uses it. Three auth methods, all first-class:

1. Claude subscription (setup-token) — Claude Max users paste a session token. No API key needed.
2. ChatGPT subscription (PKCE OAuth) — ChatGPT Plus users authenticate via browser OAuth flow. No API key needed.
3. BYO API key — Anthropic or OpenAI API keys for users who prefer direct API access.

Subscription auth is not a hack or workaround. It's the primary onboarding path. The auth wizard in fx-cli guides users through all three methods. Credentials are stored encrypted in .fawx/auth.json.

Why this matters: The biggest barrier to adoption is "I need an API key." Most people don't have one. Most people do have a ChatGPT or Claude subscription. Removing that barrier is a product decision baked into the architecture.
