Foundation Decisions

The five structural decisions that must be locked before the Citros restructure begins. Everything else evolves as we build.

5 DECISIONS + 3 QUESTIONS — ALL RATIFIED
DECISION 1

Four-Layer Separation

The system is four layers with strict boundaries. This determines the crate structure in engine/. Higher layers can be modified at runtime; the kernel cannot.

Platform Shell
Kotlin (Android) / Swift (iOS). Renders UI, reads screen, executes device actions.
android/ · ios/
↕ UniFFI (JNI / C FFI)
Loadable Layer — HOT-SWAPPABLE
Skills (WASM), strategies, config, prompt templates. Agent can modify (with permission). A/B slotted.
engine/crates/ct-loadable/
↕ Skill API / Config API
Kernel — IMMUTABLE AT RUNTIME
Loop orchestration, policy engine, permissions, budget, verification, rollback, watchdog.
engine/crates/ct-kernel/
↕ Storage API
Storage Layer
Encrypted persistent storage. Memory stores, checkpoints, audit log.
engine/crates/ct-storage/

DECISION 2

Kotlin Drives, Rust Thinks

The FFI boundary is pure request-response. No callbacks from Rust into Kotlin. No bidirectional JNI. The engine is a stateless-looking function: data in, decisions out.

Kotlin Shell
Reads screen
Captures sensors
Executes touches
Renders UI
perception data → →
Rust Engine
Runs loop
Makes decisions
Manages memory
Enforces policy
← decisions + UI specs ←
Kotlin Shell
Executes decisions
Returns results
Repeats

Bridge: UniFFI (Mozilla). Generates Kotlin + Swift bindings from Rust. Fallback: generated code is readable and forkable.

Build: Simple cargo build script + copy .so. No Gradle plugin.

RATIFIED

FFI Boundary: One Fat Call

Kotlin sends full perception snapshot, gets back all decisions for the cycle. The engine runs the full loop internally. Kotlin's job is: gather data, send it in, get decisions back, execute them. Iterate.

Why this wins: Fewer FFI crossings. Cleaner boundary. Engine is self-contained. iOS port is trivial — same API, different shell. The Rust side owns the entire loop; the platform side is just I/O.

// The entire FFI surface:
fn run_loop_cycle(perception: PerceptionSnapshot) → LoopResult

// Where:
//   PerceptionSnapshot = screen state + sensors + notifications + user message
//   LoopResult = decisions to execute + UI updates + completion status
DECISION 3

Seven-Step Recursive Loop

Every agent action follows the same cycle. The loop is a recursive function: Loop(goal, context, depth) → LoopResult. Sub-goals, evidence gathering, and recovery all spawn child loops with managed depth, scoped context, and independent checkpoints.

1 Perceive
→
2 Reason
→
3 Decide
→
4 Act
→
5 Verify
→
6 Learn
→
7 Continue
Recursion & Re-entry Points
3 → 2 Denied — intent rejected by policy/budget/permission. Re-enter Reason with denial as constraint. (Max reformulation depth: 2) 3 → ↻ Decompose — goal too complex for single action. Spawn child Loop(sub_goal, depth+1) for each sub-goal. 4 → ↻ Navigation — action requires a different screen/app. Spawn child loop to navigate there first, then resume parent. 5 → 4 Retry — verification failed, recovery strategy says retry. Re-execute same action. (Retry budget kernel-enforced) 5 → 2 Replan — verification failed, recovery says try different approach. Back to Reason with failure context. 5 → ↻ Evidence — verification inconclusive. Spawn child Loop(verify_intent, depth+1) to gather more evidence. 7 → 1 Continue — more work needed. Fresh Perceive with updated context. New reasoning cycle. 7 → 3 More intents — same reasoning produced multiple intents. Next intent enters Decide.
↻ = spawns child loop. Max depth: 3 (kernel-enforced). Child inherits parent context (read-only), gets scoped budget, checkpoints independently.

What this locks: The kernel crate implements Loop(goal, context, depth) → LoopResult with modules for each step. The recursion, re-entry paths, and depth management are kernel-level — not optional extensions. The engine's public FFI surface is run_loop(perception) → LoopResult which invokes the recursive loop internally.

DECISION 4

Four Memory Types

Memory is four distinct systems. This determines the storage crate's module structure.

Working
Conversation context, screen state, active task. The scratchpad.
Ephemeral — dies with the task. Kernel-managed.
Episodic
What happened, when, what the outcome was. Interaction history.
Persistent, time-decaying. Storage layer.
Semantic
Durable facts, preferences, identity. Things that are true.
Persistent, curated. Storage layer.
Procedural
Strategies, prompt templates, recovery patterns. How to do things.
Persistent, versionable. Loadable layer (A/B slotted).
DECISION 5

Kernel Invariants

These are the guarantees the kernel must enforce in compiled Rust. They define what goes in ct-kernel vs what's configurable. They cannot be overridden by skills, config, or the agent.

🔒 Permission registry is user-only. The agent cannot escalate its own permissions.
🔒 Policies are user-only. Action allow/deny/confirm rules can only be changed by the user.
🔒 Rollback cannot be disabled. A/B slots and watchdog are always active.
🔒 Audit trail is append-only. Neither the agent nor skills can edit or delete log entries.
🔒 Capability dropping is one-way. Skills cannot gain capabilities beyond what they were granted.
🔒 Kernel is immutable at runtime. Loop orchestration, state machine, policy engine — compiled Rust, never patchable.
🔒 Consolidation requires checkpoint. Memory dreaming cannot start without a validated atomic snapshot.
🔒 Three-gate decision. Every action passes policy → budget → permission. No gate can be skipped.

RATIFIED

Kotlin Migration: Early Move

Move Kotlin files into android/ early (PR 2-3), but keep all logic in Kotlin. Engine crates get created as stubs. The directory structure is right from the start — each subsequent PR fills in engine logic and thins the Kotlin side.

Why this wins: Structure is correct from day one. No "old layout" vs "new layout" confusion. Kotlin keeps working exactly as-is. The engine boundary is established even when the engine is empty.
