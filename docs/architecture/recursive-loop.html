Recursive Loop & Task Decomposition

The loop as a recursive function — not just for task decomposition, but for verification, navigation, recovery, and any situation where a step needs more intelligence than a single pass provides.

Why recursion: The seven-step loop is linear in specification but recursive in practice. During Reason, the agent might need to Perceive more. During Act, an action might require navigation. During Verify, inconclusive results need a fresh Perceive→Reason cycle. Task decomposition means sub-goals each run their own loop. The linear loop is a simplification that breaks on real-world tasks. Making recursion explicit is how we handle complexity without abandoning the safety model.

The Recursive Signature

Loop(goal: Goal, context: LoopContext, depth: u8) → LoopResult

where:
  Goal     = what this loop invocation is trying to achieve
  Context  = parent's working memory (read-only) + this level's working memory (read-write)
  depth    = current recursion depth (0 = top-level user request)

returns:
  LoopResult::Completed(evidence)      // Goal achieved, verified
  LoopResult::Failed(reason)           // Unrecoverable failure
  LoopResult::NeedsUser(context)       // Escalation — user input needed
  LoopResult::DepthExceeded            // Hit max recursion depth
  LoopResult::BudgetExhausted          // Hit budget limit for this sub-loop
  LoopResult::Interrupted(checkpoint)  // User cancelled or steered

Full Recursive Loop

Loop(goal, context, depth) → LoopResult:
  // KERNEL GATE: depth limit
  if depth > MAX_DEPTH:
    return LoopResult::DepthExceeded

  // Step 1: PERCEIVE
  perception = perceive(context)

  // Step 2: REASON
  plan = reason(perception, goal)

  // If pure text response (no actions needed):
  if plan.is_response_only():
    return LoopResult::Completed(plan.response)

  for sub_goal in plan.goals:
    // Step 3: DECIDE
    decision = decide(sub_goal)

    match decision:
      Denied(reason) →
        // Feed denial back, re-reason with constraint
        plan = reason(perception.with_constraint(reason), goal)
        continue  // try re-planned goals

      PendingConfirmation →
        // Surface to user, wait for approval
        approval = await_user_confirmation(sub_goal)
        if !approval: continue

      RequiresDecomposition →
        // RECURSE: sub-goal needs its own loop
        sub_result = Loop(
          sub_goal,
          context.child_scope(budget_fraction),
          depth + 1
        )
        learn(sub_result)
        if sub_result.is_failure():
          recovery = consult_recovery(sub_result)
          match recovery:
            Replan → break   // back to top of loop with failure context
            Escalate → return LoopResult::NeedsUser(context)
            Abort → return LoopResult::Failed(sub_result.reason)
        continue

      Approved(intent) →
        // Step 4: ACT
        checkpoint_state()
        result = act(intent)

        // Step 5: VERIFY
        outcome = verify(result, intent.expected_outcome)

        match outcome:
          Verified →
            // Step 6: LEARN
            learn(VerifiedOutcome { intent, result, outcome })

          Failed →
            recovery = consult_recovery(outcome)
            match recovery:
              Retry → continue  // retry this sub_goal
              GatherEvidence →
                // RECURSE: evidence-gathering sub-loop
                evidence = Loop(
                  Goal::Verify(intent),
                  context.child_scope(evidence_budget),
                  depth + 1
                )
                re_outcome = verify_with_evidence(result, evidence)
                learn(re_outcome)
              Replan → break  // back to reason with failure context
              Escalate → return LoopResult::NeedsUser(context)
              Abort → return LoopResult::Failed(reason)

          Inconclusive →
            // RECURSE: gather more evidence
            evidence = Loop(
              Goal::Verify(intent),
              context.child_scope(evidence_budget),
              depth + 1
            )
            re_outcome = verify_with_evidence(result, evidence)
            learn(re_outcome)

  // Step 7: CONTINUE or COMPLETE
  completion = evaluate_completion(goal, all_outcomes)
  match completion:
    Complete → return LoopResult::Completed(evidence)
    Continue → return Loop(goal, context.updated(), depth)  // re-perceive
    MaxSteps → return LoopResult::Completed(partial_results)

When Recursion Naturally Occurs

Task Decomposition

Trigger: The Reason step determines the goal is too complex for a single action and outputs a TaskPlan with sub-goals.

Example: "Book a trip to Denver" → [search flights, select and book flight, add to calendar, text itinerary to partner]

Recursion type: Each sub-goal gets its own Loop(sub_goal, ...) invocation. Sub-goals can depend on each other (book flight must complete before adding to calendar) or be independent (text itinerary can happen after calendar but doesn't depend on it).

Inconclusive Verification

Trigger: The Verify step can't determine if the action succeeded from available evidence.

Example: Agent tapped "Send" but can't tell if the message actually sent. → Sub-loop: take screenshot, read screen, look for confirmation indicator.

Recursion type: Evidence-gathering sub-loop. Goal is Verify(original_intent). Typically shallow (depth+1 only, no further decomposition). Uses a smaller budget allocation.

Navigation Prerequisites

Trigger: The intended action requires navigating to a different screen/app first.

Example: "Type in the search box" but search box isn't visible → Sub-loop: open app, navigate to search screen, find and tap search box.

Recursion type: Prerequisite sub-loop. The parent loop's Act step discovers it can't execute directly and spawns a navigation sub-loop. Common in mobile contexts where actions require multi-step navigation.

Recovery Escalation

Trigger: An action failed and the recovery strategy suggests an alternative approach that's complex enough to need its own planning.

Example: Tapping a button failed (element not found) → recovery: scroll to find it, try different selector, or navigate to the feature via a different path.

Recursion type: Recovery sub-loop. Similar to navigation but triggered by failure. The sub-loop's goal is to achieve the same end state through an alternative path.

Context Management Across Depth

PropertyBehaviorRationale
Context inheritance Child loops can read parent working memory. Writes are scoped to child only. Child needs to know what the parent was doing (e.g., what app is open) but shouldn't pollute parent's context with navigation details.
Result propagation Child LoopResult flows back to parent as evidence. Parent's Learn step incorporates child outcomes. The parent needs to know if the sub-goal succeeded, failed, or was escalated. But it doesn't need the full trace.
Budget partitioning Parent allocates a budget fraction to each child. Child cannot exceed its allocation. Unspent budget returns to parent. Prevents runaway sub-loops from consuming the entire task budget. Forces the agent to allocate resources thoughtfully.
Independent checkpointing Each depth level checkpoints independently. Crash at depth 2 → resume at depth 2 without replaying depth 0-1. Crash recovery proportional to the interrupted work, not the total task. Essential for long multi-step tasks.
Memory access All depths share the same episodic/semantic memory stores for retrieval. Writes from child loops are gated by the same policies. The agent's knowledge shouldn't be siloed by recursion depth. A fact learned at depth 2 is available at depth 0.

Sub-Goal Dependencies

When the Reason step produces a TaskPlan, sub-goals can have dependency relationships:

TaskPlan {
  sub_goals: Vec<SubGoal>,
  dependencies: Vec<Dependency>,  // (a, b) means a must complete before b starts
}

// Execution strategy:
// 1. Topological sort of dependency graph
// 2. Independent sub-goals at the same level CAN run in sequence
//    (parallel execution is a future optimization)
// 3. If a dependency fails, all dependents are skipped
//    and the planner is re-consulted with failure context
Dependency typeExampleOn failure
Hard dependency Must book flight before adding to calendar (need flight details) Dependent sub-goals skipped. Re-plan with failure context.
Soft dependency Should add to calendar before texting itinerary (nicer), but can text without calendar entry Dependent sub-goals continue. Failure noted in context.
Independent Booking hotel and booking flight can happen in either order No impact on other sub-goals.

Progress Tracking & UI

The platform shell renders progress for multi-step tasks. The kernel emits progress events that the UI consumes.

// Kernel emits:
ProgressEvent {
  task_id:     TaskId,
  plan:        TaskPlan,      // Full plan structure (for rendering)
  current:     SubGoalId,     // Which sub-goal is active
  completed:   Vec<(SubGoalId, Outcome)>,  // Finished sub-goals
  depth:       u8,            // Current recursion depth
  budget_used: f32,           // Percentage of total budget consumed
}

// UI renders (example):
// 📋 Send birthday message to Mom
//   ✅ Open Messages
//   ✅ Navigate to Mom's conversation
//   🔄 Compose and send message...
//   ⏳ Verify delivery

Depth visibility: The UI shows top-level sub-goals. Deeper recursion (evidence gathering, navigation prerequisites) is hidden by default — it's implementation detail. The user can expand to see full depth if they want.

Recursion Invariants

🔒 Maximum recursion depth is kernel-enforced. Default: 3. The agent cannot increase this. If a task requires deeper recursion, it's escalated to the user as "too complex for automatic execution."
🔒 Budget partitioning is kernel-enforced. Child loops cannot exceed their allocated budget. A runaway sub-loop hits its budget wall, not the parent's.
🔒 Each recursion level checkpoints independently. Crash recovery resumes at the exact depth and step where the crash occurred.
🔒 Child writes are scoped. Child loops can read parent working memory but write only to their own scope. Prevents sub-loops from corrupting parent state.
🔒 All policy gates apply at every depth. A sub-loop at depth 3 passes through the same Decide gates (policy, budget, permission) as depth 0. Recursion doesn't bypass safety.

State Machine with Recursion

The kernel maintains a stack of state machines, one per recursion level. Each level follows the standard 7-step state machine independently.

LoopStack {
  frames: Vec<LoopFrame>,  // frames[0] = top-level, frames[n] = deepest active
  max_depth: u8,
}

LoopFrame {
  goal:           Goal,
  state:          LoopState,   // Idle | Perceiving | Reasoning | Deciding | ...
  working_memory: WorkingMemoryFrame,
  checkpoint:     Option<Checkpoint>,
  budget:         Budget,
  parent_context: ReadOnlyRef<WorkingMemoryFrame>,  // read-only view of parent
}

// State transitions happen within a single frame.
// Recursion pushes a new frame. Return pops it.
// The kernel always processes the deepest (most recent) frame.
Design principle: Recursion is not a feature added to the loop — it's a fundamental property of how intelligent agents solve real problems. By making it explicit in the architecture, we get managed depth (safety), budget partitioning (resource control), independent checkpointing (crash resilience), and scoped context (clean separation) — all enforced by the kernel, all impossible to bypass.
