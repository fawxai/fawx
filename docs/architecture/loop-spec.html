Agentic Loop Specification

The execution cycle that drives every agent action — from perception through verification to memory

Loop Overview

Every agent action follows the same seven-step cycle. The kernel orchestrates the loop; loadable strategies influence decisions within it; the platform shell provides perception and executes physical actions.

State Machine

Idle → Perceiving → Reasoning → Deciding → Executing → Verifying → Learning → Continue / Complete

The Seven Steps

1

Perceive

Platform Shell Kernel Storage

What the agent knows right now. The platform shell provides raw perception data. The kernel assembles it into a coherent context snapshot. Memory retrieval (loadable strategy) enriches the snapshot with relevant past context.

Inputs: Screen state, sensor data (battery, network, location), notification queue, conversation history, user message

Memory interaction: Retrieve relevant episodic + semantic memories. Working memory (conversation context) is already present. Retrieval strategy is loadable — the kernel calls it but doesn't dictate how it searches.

Output: PerceptionSnapshot — structured bundle of everything the agent knows at this moment

2

Reason

LLM Inference Loadable

What should the agent do? The perception snapshot is formatted into a prompt (loadable prompt template + strategy), sent to the LLM (model selection is loadable config), and the response is parsed into typed intents.

Inputs: PerceptionSnapshot, active prompt template, model routing config

Process:

1. Prompt construction — loadable strategy assembles system prompt + context + user message
2. Model selection — loadable config picks provider and model
3. LLM inference — kernel manages the API call, timeout, retry
4. Response parsing — extract tool calls as typed Intent[]

Output: Intent[] — typed action intents with expected evidence requirements. If no intents (pure text reply), skip to step 7 (Complete).

3

Decide

Kernel — Policy Engine Kernel — Budget Enforcer Kernel — Permission Registry

Is the agent allowed to do this? Every intent passes through three kernel-enforced gates. No loadable code can bypass these.

Gate 1 — Policy: Does this intent match an allow/deny/confirm rule? If deny → reject intent, return to Reason with constraint. If confirm → surface to user via platform shell, wait for approval.

Gate 2 — Budget: Can the agent afford this action? Check token budget, API cost estimate, task cost accumulator. If over budget → reject with budget exhaustion reason.

Gate 3 — Permission: Does the intent modify a protected layer? Check permission registry. If the layer is locked or requires approval → gate accordingly.

Output: Decision per intent — Approved, Denied(reason), or PendingConfirmation

On Denial

Denied intents are fed back to the Reason step with the denial reason as context. The agent gets a chance to reformulate. This is not a retry — it's a new reasoning step with additional constraints. Max reformulation depth is kernel-configured (default: 2).

4

Act

WASM Skill Platform Shell Kernel — State Machine

Execute the approved intent. The kernel dispatches the intent to the appropriate skill (WASM module) or platform capability. The state machine transitions to Executing.

Process:

1. Kernel resolves intent → skill (tool registry lookup)
2. Kernel checkpoints state (pre-action) for crash recovery
3. Skill executes — may call platform shell (e.g., tap element, read screen)
4. Skill returns ActionResult with evidence artifacts
5. Platform shell provides post-action state (screen content, etc.)

Inputs: Approved Intent, pre-action state

Output: ActionResult — success/failure indicator + evidence artifacts (screenshots, element state, page content, etc.)

5

Verify

Kernel — Verification Engine Kernel — State Machine

Did it actually work? The tool said "success" — prove it. The verification engine checks evidence artifacts against the intent's expected outcomes. This is kernel-enforced, not optional.

Checks:

• Artifact contracts — does the evidence match what "done" looks like for this intent type?
• Screen state diff — did the screen actually change in the expected way?
• Side-effect guards — did the action claim success without evidence of the side effect?

Outcomes:

Verified → state transitions to Learning
Failed → recovery strategy consulted (loadable), may retry or escalate
Inconclusive → additional evidence gathering (e.g., take screenshot, re-read screen)

Retry / Recovery

On failure, the kernel consults the active recovery strategy (loadable). The strategy may recommend: retry same action, try alternative approach, escalate to user, or abort. Retry budget is kernel-enforced (max retries per intent, max consecutive failures across intents). Recovery strategies can suggest, but cannot override the budget.

6

Learn

Kernel — Memory Policy Memory Strategies Storage Layer

What should the agent remember? After each action, the agent updates its memory systems. What gets stored is gated by kernel-enforced memory policies (user-controlled). How it's stored and indexed is determined by loadable memory strategies.

Updates by memory type:

Working memory: Append action result to conversation context. Run compaction if context window is approaching limits (compaction strategy is loadable).

Episodic memory: Record what happened — intent, action, outcome, timestamp. Gated by retention policy (user-controlled: "don't remember browsing history").

Semantic memory: Extract durable facts if applicable ("user's dentist is Dr. Smith on Oak St"). Gated by privacy policy. Indexing strategy is loadable.

Procedural memory: If a novel strategy worked (or a standard strategy failed repeatedly), propose updating the strategy. This goes through the A/B slot lifecycle with permission gating.

7

Continue or Complete

Kernel — State Machine

Are there more intents? The kernel evaluates whether the task is complete.

Continue: More intents from the same reasoning step → back to step 3 (Decide) for next intent. All intents processed but task not done → back to step 1 (Perceive) for fresh context + new reasoning.

Complete: Goal achieved (verified), max steps reached, user cancelled, or unrecoverable failure. Checkpoint final state, respond to user.

Completion gate: The kernel checks whether the final text response actually addresses the user's request (completion contract). A response that says "I opened the app" when the user asked "send a message to Chris" is not complete — the loop continues.

↺ Loop back: Step 7 → Step 1 (fresh perception for next action cycle) or Step 7 → Step 3 (more intents from same reasoning)

Interrupt Handling

The loop must handle external events at any point — user messages, cancellation, accessibility loss, budget exhaustion. The kernel checks for interrupts at defined boundary points.

User Steer (new message mid-task)

Checked at: pre-batch boundary, post-action boundary

User sent a new message while the agent is working. The message is delivered to conversation history. The current action completes, then the loop re-enters Perceive with the new context.

Action: Complete current action → deliver steer → re-Perceive

User Cancel

Checked at: every boundary point + mid-action for composite actions

User pressed Stop. The kernel halts execution at the nearest atomic boundary and assesses rollback.

Atomic actions (single tap, keystroke, API call): Already in-flight at the OS level — let it land, but do not proceed to the next step. No rollback needed.
Composite actions (multi-step: typing a paragraph, filling a form): Stop at the current atomic boundary. Roll back to pre-action checkpoint if partial state is worse than pre-action state.
Side-effect actions (message sent, form submitted, purchase made): If the side effect already fired, it cannot be recalled — log what completed and inform the user. If not yet fired, cancel prevents it.

After halt: restore from pre-action checkpoint (unless the action completed atomically and cleanly). Exit with cancelled reason. Checkpoint final state for potential resume.

Action: Halt at atomic boundary → assess rollback → restore checkpoint if partial → inform user what completed → exit

Accessibility Lost

Checked at: post-action boundary

The perception source detached (shell disconnected, service killed). The agent cannot perceive or act on the environment. Loop exits with perception_lost.

Action: Exit loop → notify user → wait for re-attachment

Budget Exhausted

Checked at: Decide step (Gate 2)

The task has exceeded its cost budget. No more API calls allowed. The agent responds with what it has so far and explains the budget limit.

Action: Exit loop with partial result → explain budget limit

Memory System Interactions

Each loop step interacts with memory differently. The kernel enforces what CAN be stored; loadable strategies determine HOW.

Working Memory

📖 Read at: every step (it IS the conversation context)
✏️ Written at: Step 4 (action results appended), Step 6 (compaction)

In-kernel, ephemeral. Compaction strategy (loadable) decides what to keep vs summarize vs drop when approaching context limits.

Policy: Kernel manages size. No user gate needed (ephemeral).

Episodic Memory

📖 Read at: Step 1 (Perceive — retrieve relevant past interactions)
✏️ Written at: Step 6 (Learn — record what happened)

Persistent, encrypted. Time-decaying relevance. Used for "last time you asked about X" and task outcome history.

Policy: User controls retention categories. "Forget all browsing." "Keep task outcomes for 30 days."

Semantic Memory

📖 Read at: Step 1 (Perceive — enrich context with known facts/preferences)
✏️ Written at: Step 6 (Learn — extract durable facts from interactions)

Persistent, indexed, curated. "Joe's timezone is Mountain." "Preferred airline is United." User can view, edit, and delete entries.

Policy: User controls what categories are stored. Can review and delete any entry.

Procedural Memory

📖 Read at: Step 2 (Reason — which strategy to use), Step 5 (Verify — recovery approach)
✏️ Written at: Step 6 (Learn — propose strategy updates via A/B slots)

Persistent, versionable. The agent's learned behaviors — prompt strategies, recovery patterns, tool preferences. Changes go through A/B slot lifecycle.

Policy: Agent proposes → user approves (or auto-approve if permission is "free").

What the Kernel Owns vs What's Loadable

Concern Kernel (compiled, immutable) Loadable (hot-swappable)
Loop orchestrationStep sequencing, state transitions, boundary checks—
PerceptionAssembling the snapshot, managing working memoryMemory retrieval strategy, attention prioritization
ReasoningManaging LLM call, timeout, retryPrompt templates, model selection, strategy choice
DecisionPolicy enforcement, budget enforcement, permission checks—
ActionDispatch, checkpointing, state transitionsTool implementations (WASM skills)
VerificationArtifact contract checking, retry budget enforcementRecovery strategies, verification heuristics
LearningMemory policy enforcement, write gatingWhat to extract, how to index, compaction strategy
CompletionCompletion contract checking, final state checkpointCompletion heuristics (what counts as "done")
The invariant: The kernel guarantees the loop runs correctly, safely, and within policy. Loadable strategies make the loop run well — they determine intelligence, not safety. If every loadable module were replaced with a no-op, the loop would still run (doing nothing useful, but never doing anything harmful).

Loop Extensions

Six extensions build on the core loop. Each is fully specified in its own document. Together they transform the base loop from a reactive executor into a proactive, self-improving, recursive intelligence.

A

Anticipation Loop — Proactive Perception

Background perception during idle. Trigger classification, three-tier suggestions (time-critical / contextual / passive), power management, fatigue prevention. The agent offers help before you ask — carefully.
B

Expectation-Verification — Sensory Prediction

Agent predicts post-action state before acting, then compares prediction to reality. Two-pass verification (contract + prediction). Prediction accuracy feeds learning. Catches subtle failures that pass static contracts.
C

Recursive Loop & Task Decomposition

Loop(goal, context, depth) — the loop as a recursive function. Task decomposition, evidence gathering, navigation prerequisites, recovery escalation. Managed stack, scoped context, independent checkpointing, budget partitioning.
D

Memory Consolidation ("Dreaming")

Six-phase consolidation: Score → Consolidate → Strengthen → Prune → Identity → Audit. Pre-dream checkpoints, importance-weighted decay, episodic→semantic promotion, dream reports. The agent gets smarter while you sleep.
E

Memory-Augmented Intelligence

Four ascending layers: RAG → Behavioral conditioning → Learned retrieval attention (MANN) → On-device fine-tuning. Each makes the agent measurably smarter. Architecture supports all four from day one.
Note: Task decomposition (originally proposed as a separate extension) is subsumed by the recursive loop — decomposition is one of four recursion triggers, not a standalone feature. See Recursive Loop for the full treatment.
