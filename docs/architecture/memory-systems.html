Memory Systems

Four memory types with distinct lifecycles, storage models, and permission boundaries. Memory feeds the loop. The loop feeds memory. Together they make the agent learn.

Core principle: Memory is not a database — it's a learning system. Each memory type serves a different cognitive function, has a different lifecycle, and is controlled differently. The separation is architectural, not cosmetic. Mixing them would compromise both safety and intelligence.

The Four Memory Types

Working Memory

Kernel-managed

What it holds: Current conversation context, screen state, active task state, pending intents, action results in flight. Everything the agent needs right now to continue its current task.

Lifecycle: Ephemeral — created when a task starts, dies when the task completes (or is abandoned). Never persisted to storage. Never checkpointed for dreaming.

Size management: Bounded by the context window. When approaching limits, the kernel invokes a compaction strategy (loadable) that summarizes older entries, drops low-relevance content, and preserves high-signal items. The kernel enforces the size limit; the strategy decides what to compact.

Analogy: A whiteboard in front of you during a meeting. When you run out of space, you erase the least important stuff. When the meeting ends, the board is wiped.

Episodic Memory

Storage layer Kernel-gated writes

What it holds: What happened, when, and what the outcome was. Task records: intent → action → result → verification outcome → timestamp. Interaction history: what the user asked, what the agent did, whether it worked.

Lifecycle: Persistent with time-decay. Every episode has an importance score that decays over time unless reinforced by retrieval or outcome contribution. Entries below the importance threshold are candidates for pruning during dreaming cycles.

Storage: Encrypted, in the storage layer. Each entry includes: content, timestamp, importance score, embedding vector, provenance metadata, category tags.

Retrieval: During the Perceive step, the retrieval strategy (loadable) searches episodic memory for entries relevant to the current context. Importance-weighted: higher-scoring entries get retrieval priority within the context budget.

Analogy: Your diary. You remember what happened each day, but older entries fade unless they were important or you revisited them.

Semantic Memory

Storage layer Kernel-gated writes

What it holds: Durable facts, preferences, knowledge. Things that are true independent of when they were learned. "Joe's timezone is Mountain." "Thai Palace is on 5th Street." "The dentist appointment is with Dr. Smith." Also: the identity profile (see User Identity Model).

Lifecycle: Persistent, curated. Not time-decaying by default (facts don't expire just because they're old). Can be updated when new information contradicts old facts. Can be deleted by the user at any time.

Storage: Encrypted, indexed for efficient retrieval. Each entry includes: content, confidence score, embedding vector, provenance chain (which episodic memories led to this fact), category, last-verified timestamp.

Retrieval: During Perceive, semantic entries relevant to the current context are loaded. The identity profile is always loaded (kernel-enforced). Other entries use the same importance-weighted retrieval as episodic, but without time-decay bias.

Promotion from episodic: During dreaming, consolidation detects patterns in episodic memory and promotes them to semantic facts. "Joe ordered from Thai Palace 4 times this month" (episodic pattern) → "Joe's frequent Thai restaurant is Thai Palace" (semantic fact). Promotion requires a provenance chain — every semantic fact must trace back to its source episodes.

Analogy: Your knowledge base. You know facts without remembering exactly when you learned them, but you trust them because they've been reinforced by experience.

Procedural Memory

Loadable layer (A/B slotted) Kernel-gated modifications

What it holds: How to do things. Reasoning strategies, recovery strategies, prompt templates, compaction strategies, memory retrieval strategies, tool preferences, navigation patterns. The agent's learned behaviors, not its learned facts.

Lifecycle: Persistent, versionable. Changes go through the A/B slot lifecycle — proposed by the agent (or by the system during dreaming), validated by the kernel, gated by the permission registry, deployed with automatic rollback on failure.

Storage: In the loadable layer, not the storage layer. Procedural memory is the loadable configuration — strategies, templates, and configs that shape how the loop executes.

Learning: During the Learn step, if a strategy consistently underperforms (repeated verification failures, high retry rates), the agent may propose an updated strategy. During dreaming, consolidated outcome patterns can trigger strategy evolution proposals.

Analogy: Muscle memory + habits. You don't think about how to ride a bike — you just do it. But the way you ride was learned through practice, and it can be updated if you learn a better technique.

Memory Entry Schemas

Episodic Entry

EpisodicEntry {
  id:              UUID,
  timestamp:       Instant,
  content:         String,              // What happened (structured or natural language)
  category:        Category,            // task_outcome | interaction | observation | error
  intent:          Option<Intent>,      // The intent that produced this episode (if action-derived)
  outcome:         Option<Outcome>,     // success | failure | partial | inconclusive
  importance:      ImportanceScore {
    retrieval_freq:      f32,           // How often retrieved during Perceive
    outcome_contribution: f32,          // Did this contribute to successful tasks?
    recency:             f32,           // Time-decaying score
    composite:           f32,           // Weighted combination — the actual sorting key
  },
  embedding:       Vec<f32>,           // For vector similarity retrieval
  provenance:      Provenance {
    task_id:        Option<TaskId>,    // Which task produced this
    loop_step:      LoopStep,          // Which step generated it (usually Learn)
    source_type:    SourceType,        // user_message | action_result | observation | consolidation
  },
  retention_class: RetentionClass,     // Determines decay curve and pruning eligibility
  tags:            Vec<String>,        // User-defined + auto-classified
}

Semantic Entry

SemanticEntry {
  id:              UUID,
  content:         String,              // The fact ("Joe's timezone is Mountain Time")
  category:        Category,            // identity | preference | fact | relationship | location
  confidence:      f32,                 // 0.0–1.0, updated on reinforcement or contradiction
  embedding:       Vec<f32>,
  provenance:      SemanticProvenance {
    source_episodes: Vec<EpisodicId>,  // Which episodes led to this fact
    first_inferred:  Instant,
    last_reinforced: Instant,
    reinforcement_count: u32,
  },
  contradictions:  Vec<Contradiction>, // Past contradictions and how they were resolved
  last_verified:   Instant,            // When this fact was last confirmed true
  user_editable:   bool,               // Always true — user can always modify
}

Retrieval Architecture

Memory retrieval happens at the Perceive step. The kernel calls the retrieval strategy (loadable); the strategy returns ranked memory hits within a token budget.

Retrieval Pipeline

Current context → Embed query → Candidate search → Importance-weighted rank → Token budget fit → MemoryHits[]
Pipeline stageOwnerDetails
Embed query Loadable Convert current context (user message + task state) into an embedding vector. Strategy chooses: full context embedding, keyword extraction, or hybrid.
Candidate search Storage Vector similarity search across episodic + semantic stores. Returns top-K candidates by cosine similarity. K is configurable (default: 50).
Importance-weighted rank Loadable Re-rank candidates using composite importance score. Balances relevance (embedding similarity) with importance (retrieval frequency, outcome contribution, recency). Weighting function is loadable.
Token budget fit Kernel Pack ranked results into the memory portion of the context window. Budget is kernel-determined based on total context limits minus conversation + system prompt. Truncates at the budget boundary.

Special Retrieval Rules (Kernel-Enforced)

🔒 Identity profile is always loaded. The user identity profile (semantic memory) is loaded into every PerceptionSnapshot regardless of relevance scoring. The agent never reasons without knowing who it serves.
🔒 Retrieval is logged. Every retrieval (which entries were returned, which were considered but excluded) is recorded. This data feeds importance score updates: retrieved entries get their retrieval_freq incremented.

Write Gating

Writing to persistent memory (episodic, semantic) is gated by kernel-enforced policies. The agent proposes writes during the Learn step; the kernel checks each against the user's memory policy before committing.

GateChecksOn failure
Category filter Is this memory in an allowed category? User can block entire categories ("never store health data", "don't remember browsing history"). Write silently dropped. Logged in audit trail.
Privacy filter Does this memory contain PII or sensitive data that the user hasn't consented to storing? Checked against the privacy policy in the permission registry. Write blocked. Logged. Agent informed (can try reformulating without sensitive data).
Deduplication Is this substantially similar to an existing entry? Vector similarity check against recent entries to prevent redundant storage. Existing entry's importance score updated (reinforcement). New entry discarded.
Provenance requirement Does the write have a valid provenance chain? Semantic entries must trace to source episodes. Fabricated memories are rejected. Write rejected. Logged as anomaly.

Memory × Loop Interactions

Loop StepReads fromWrites toNotes
1. Perceive Working (always present), Episodic (retrieval), Semantic (retrieval + identity) Working (assembles snapshot) Retrieval strategy is loadable. Identity always loaded.
2. Reason Working (snapshot), Procedural (active strategy) — Procedural memory determines prompt template and model selection.
3. Decide — — Decision gates are kernel policy, not memory-dependent.
4. Act Working (intent details) Working (action results appended) Pre-action checkpoint saved for crash recovery.
5. Verify Working (action results + expected outcomes), Procedural (recovery strategy) Working (verification outcome) Recovery strategy from procedural memory consulted on failure.
6. Learn Working (full cycle results) Episodic (task outcome), Semantic (extracted facts), Procedural (strategy proposals) All writes gated by kernel policy. Procedural changes go through A/B slots.
7. Continue Working (task completion state) Working (compaction if continuing) On completion: working memory is available for episodic extraction, then discarded.

Graph Index (Design Now, Build Later)

Semantic memory entries are flat by default — connected only by embedding similarity. But real-world knowledge is relational: people know people, places contain things, events involve participants. A graph index over semantic memory captures these relationships and enables traversal, not just search.

Architecture decision: The graph is an index structure over semantic memory, not a fifth memory type. The four memory types stay the same. The graph is a complementary retrieval path — vector similarity for "what's relevant?" and graph traversal for "what's connected?"

Schema Additions

SemanticEntry {
  // ... existing fields ...

  // NEW: Entity and relationship support
  entities:       Vec<EntityRef>,        // Named entities extracted from this entry
  relationships:  Vec<Relationship>,     // Connections to other entities
}

EntityRef {
  id:             EntityId,              // Stable identifier
  name:           String,                // "Dr. Smith", "Thai Palace", "Mom"
  entity_type:    EntityType,            // person | place | organization | event | concept
  aliases:        Vec<String>,           // ["Dr. Robert Smith", "the dentist"]
}

Relationship {
  subject:        EntityRef,             // "Joe"
  predicate:      String,                // "visits", "works_at", "prefers", "lives_near"
  object:         EntityRef,             // "Dr. Smith"
  confidence:     f32,                   // How confident we are in this relationship
  source_episodes: Vec<EpisodicId>,      // Provenance — which episodes established this
  first_observed: Instant,
  last_reinforced: Instant,
  strength:       f32,                   // Reinforcement count / recency-weighted
}

How the Graph Gets Built

PhaseWhenWhat happens
Entity extraction Learn step (every loop cycle) When episodic entries are created, extract named entities (NER). Tag the episode with entity refs. Lightweight — no LLM needed, rule-based or small local model.
Relationship extraction Dreaming (Phase 2 — Consolidation) When episodic clusters are promoted to semantic facts, extract relationships between entities in the cluster. Requires LLM inference (same call as fact extraction). Relationships stored on the semantic entry.
Graph index update After any semantic write The graph index is updated whenever a semantic entry with relationships is created or modified. The index is a derived structure — it can always be rebuilt from semantic entries.
Reinforcement Dreaming (Phase 3 — Strengthen) Relationships that appear across multiple episodes get their strength score boosted. Relationships not reinforced decay (but slower than episodic entries — relationships are more durable than events).

Graph-Enhanced Retrieval

During Perceive, the retrieval pipeline gains a second path:

Context → Extract entities → Graph neighborhood → Merge with vector results → Enriched MemoryHits[]

Vector path (existing): embed context → similarity search → importance-weighted rank → results.
Graph path (new): extract entities from context → traverse 1-2 hops in graph → return connected entities + their semantic entries → merge with vector results, dedup, re-rank within token budget.

The graph path catches things vector similarity misses. "Message from Dr. Smith" → graph expands to: Dr. Smith works at Oak Street Dental, appointment is next Tuesday, last visit was January. Embedding similarity alone might surface "Dr. Smith" entries but miss the connected context.

What This Enables

CapabilityWithout graphWith graph
Social graph Flat semantic entries: "Mom's name is Sarah." No relationship structure. Full relationship model: Sarah → mother_of → Joe, Sarah → lives_in → Denver, Joe → texts → Sarah (weekly, Sundays).
Multi-hop reasoning "Where does Joe's dentist work?" requires the right entry to be retrieved by embedding similarity. Traverse: Joe → visits → Dr. Smith → works_at → Oak Street Dental. Answer found by structure, not search luck.
Context enrichment Retrieved memories are independently relevant entries. Retrieved memories include relationship context: not just "Dr. Smith" but the full neighborhood of connected facts.
Dreaming quality Pattern detection via embedding clustering. Pattern detection via graph structure: "these 4 episodes all involve the same entity cluster" is a stronger signal than embedding proximity.

Build Timeline

Phase 1: Entity extraction during Learn step (lightweight NER). Entities stored but graph index not yet active.
Phase 2: Relationship extraction during dreaming. Graph index built. Graph-enhanced retrieval activated alongside vector path.
Phase 3: Graph visualization for the user (like TribalMemory's Cytoscape.js UI) — "see what your agent knows about you and your world."

Memory Transitions

Memory doesn't just sit in stores — it moves between them. Each transition has explicit criteria, quality gates, and a kernel/loadable split. Getting these transitions right is what makes the difference between an agent that accumulates data and one that actually learns.

Transition 1: Working → Episodic (at the Learn step)

When a task ends, working memory dies. The Learn step extracts what's worth keeping before that happens. This is a classification problem — most of working memory is transient context that has no long-term value.

✓ Should become an episode

• Any action that went through Decide→Act→Verify (the agent did something — record intent, action, outcome)
• User statements with semantic content ("I prefer window seats", "my dentist is Dr. Smith")
• User corrections and overrides (always — strongest learning signal)
• Novel information that contradicts or extends existing semantic memory
• Notable failures (learning from mistakes > recording successes)
• Task completion outcomes (goal achieved? partially? what was the gap?)

✗ Should NOT become an episode

• Routine acknowledgments ("ok", "thanks", "got it")
• Information already in semantic memory with high confidence (reinforce existing entry via dedup, don't create new episode)
• Intermediate navigation steps (opened app, tapped button) unless they failed or were novel
• Working memory compaction artifacts (summaries of older context)
• Repeated re-reads of the same screen content
• System-level noise (retry attempts, timeout recoveries)

ConcernLayerDetails
Extraction runs Kernel The Learn step is never skipped. The kernel guarantees extraction is attempted for every completed loop cycle.
What's worth saving Loadable The extraction strategy classifies working memory entries as save/discard. Evolves via A/B slots as the agent learns what's worth remembering.
Write gates Kernel All episodic writes pass through category filter, privacy filter, dedup check, and provenance requirement. No extraction strategy can bypass these.
Dedup on write Kernel + Loadable Kernel runs dedup check on every write. Loadable strategy sets the similarity threshold. If a near-duplicate exists, the existing entry is reinforced (importance boost) instead of creating a new entry.

Transition 2: Episodic → Semantic (during dreaming Phase 2)

This is where patterns become facts. The consolidation engine detects recurring themes across episodes and promotes them to durable semantic entries. Quality matters more than quantity — a bad semantic fact (wrong preference, incorrect relationship) actively degrades the agent's behavior.

Promotion Criteria

CriterionThresholdRationale
Frequency Pattern appears across ≥3 episodes (configurable) One Thai Palace order isn't a pattern. Four is. Prevents coincidences from becoming facts.
Consistency Episodes agree (no contradictions within the cluster) If 3 episodes say "prefers window seats" and 1 says "chose aisle," that's weaker than unanimous. Consistency score feeds confidence.
Importance Source episodes have above-median composite importance Patterns from important episodes (frequently retrieved, contributed to successes) are more likely to be real signals.
Novelty No existing semantic fact covers this pattern If it's already a semantic fact, reinforce the existing entry (boost confidence, extend provenance) rather than duplicating.

Quality Gates (preventing bad promotions)

🔒 Provenance chain required. Every semantic fact must trace to its source episodes. The kernel rejects promotions without provenance. No facts without evidence.
🔒 Contradiction detection. If a proposed fact contradicts an existing semantic entry, the kernel does not blindly overwrite. It compares evidence strength (episode count, importance scores, recency). The better-supported fact wins. The loser gets confidence reduced, not deleted. The contradiction is logged in the audit trail.
🔒 Confidence calibration. New semantic facts from inference start at moderate confidence (~0.6), never high. They earn high confidence through reinforcement by future episodes. This prevents a single dream cycle from creating "facts" that are actually coincidences.
🔒 Explicit always beats inferred. A user's explicit statement ("I'm vegetarian") immediately creates a high-confidence semantic fact (0.95). Inferred patterns always start moderate (0.6). One explicit correction outweighs 50 inferred episodes. The kernel enforces this asymmetry.
ConcernLayerDetails
Pattern detection Loadable Clustering algorithm, frequency thresholds, what constitutes a "pattern." Evolves via A/B slots.
Fact extraction (LLM) Loadable Summarizing episode clusters into fact statements. Prompt strategy is loadable.
Provenance enforcement Kernel Every proposed fact must have a valid provenance chain. No exceptions.
Contradiction detection Kernel Kernel detects conflicts between proposed and existing facts. Resolution strategy is loadable (recency-biased? frequency-biased?), but detection is mandatory.
Confidence bounds Kernel New inferred facts capped at 0.6. Explicit user statements start at 0.95. These bounds are kernel-enforced.

Transition 3: Episodic → Procedural (behavioral learning)

The subtlest transition — when does the agent learn a new behavior, not just a new fact? Procedural updates change how the agent acts, not what it knows. They carry higher risk (a bad strategy degrades all future tasks of that type) and go through the full A/B slot lifecycle.

What triggers a procedural proposal

TriggerSignalExample
Consistent underperformance Verification failure rate for an action type exceeds threshold (default: 40% over ≥5 attempts) Text input via accessibility keeps failing → propose switching to clipboard-paste strategy
Recovery success A novel approach (from recovery strategy) succeeded where the standard approach failed ≥2 times Standard element selector failed, but XPath worked 3 times → propose XPath as primary for this app
User correction pattern User corrects the same type of output ≥3 times with consistent direction User keeps saying "too formal" → propose communication style adjustment in prompt template
Dreaming pattern detection Consolidation identifies a cluster of failures with a common cause Multiple tasks failed because the agent didn't check if the app was in foreground → propose pre-action foreground check

Process: The proposal goes through the full A/B slot lifecycle — written to pending slot, validated by kernel (schema, capabilities), permission-gated (user approval if required), deployed alongside the existing strategy, compared over N tasks, promoted or rolled back based on verified success rate improvement. See Kernel Architecture — A/B Slots.

ConcernLayerDetails
Trigger detection Kernel Kernel tracks per-action-type failure rates, recovery success rates, and correction frequencies. These metrics are always collected.
Proposal formulation Loadable What the new strategy should be. Requires LLM inference to analyze failure patterns and propose alternatives.
A/B lifecycle Kernel Full slot lifecycle with automatic rollback. Kernel-enforced, cannot be bypassed.
Comparison metrics Kernel Statistical significance threshold before promoting new strategy. Minimum N tasks evaluated.

Transition Summary

The principle: Each transition has ascending stakes. Working→Episodic is cheap (recording what happened — low risk, high volume). Episodic→Semantic is moderate (distilling patterns into facts — medium risk, requires quality gates). Episodic→Procedural is expensive (changing behavior — high risk, requires A/B validation). The quality gates and approval requirements scale with the stakes. The kernel enforces this escalation — loadable strategies cannot skip gates or bypass the A/B lifecycle.
