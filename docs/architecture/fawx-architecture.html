Fawx Architecture

A personal agentic engine — designed around a recursive agentic loop, structured memory, strict safety kernel, and deep personalization. TUI-first. Desktop-first with GPU. Shells are replaceable.

🧠 The agent is personal. It serves one human. It learns their identity, taste, communication style, preferences, and needs. Every perception, plan, action, and judgment is shaped by who that human is — not by generic defaults.
🔒 Safety is architecture, not trust. The kernel enforces policy, permissions, budgets, and rollback. The agent operates within boundaries it cannot modify. Loadable intelligence makes the agent useful; compiled invariants make it safe.
🔄 The loop is recursive. Real tasks require decomposition, sub-loops, and evidence gathering. The agentic loop calls itself — with managed depth, scoped context, and independent checkpointing at every level.
💤 Memory is alive. The agent dreams — consolidating experiences into knowledge, strengthening what matters, pruning what doesn't. Memory isn't storage; it's a learning system that makes the agent better at serving its human over time.
🦀 Desktop-first with GPU. Everything built today — engine, memory, skills, protocol layer — runs unchanged as new shells appear. TUI validates the loop; Android/iOS/desktop render UISpec. No throwaway work.

Post-Dream Health Check

After consolidation, the kernel validates memory integrity before releasing the checkpoint safety net:

• Entry count sanity — did consolidation delete an anomalous percentage of entries? (Threshold: user-configurable, default 30%)
• Index consistency — do all index entries point to valid memory entries? No orphaned references.
• Semantic coherence — do newly promoted semantic facts have valid provenance chains to source episodic memories?
• Identity stability — did any identity dimension confidence score change by more than the anomaly threshold in a single cycle?
• Storage integrity — checksums on all encrypted stores match expected values.

Design principle: A/B slots protect the loadable layer (config, skills, strategies). The checkpoint protocol protects the storage layer (memories, identity, learned knowledge). Together they guarantee: any self-modification the system performs is recoverable. The system never breaks. The agent never forgets — unless the user tells it to.

Kernel Invariants

These properties are guaranteed by compiled Rust in the kernel. They cannot be overridden by skills, config, or the agent.

🔒 Permission registry is user-only. The agent cannot escalate its own permissions. No skill, config change, or prompt injection can modify what the agent is allowed to do.
🔒 Policies are user-only by default. Action policies (allow/deny/confirm rules) can only be changed by the user. The agent operates within the policy, never modifies it.
🔒 Rollback cannot be disabled. The A/B slot mechanism and watchdog timer are always active. No API exists to turn them off.
🔒 Audit trail is append-only. Every modification attempt (approved or denied) is logged. Neither the agent nor skills can edit or delete log entries.
🔒 Capability dropping is one-way. A skill granted fewer capabilities than it requests can never gain more. Escalation is structurally impossible.
🔒 The kernel is immutable at runtime. The loop orchestrator, state machine, policy engine, and all enforcement mechanisms are compiled Rust. They cannot be modified, patched, or replaced while the system is running.
🔒 Consolidation cannot proceed without a validated checkpoint. Before any memory consolidation ("dreaming") cycle, the kernel creates and verifies an atomic snapshot of all persistent memory stores. If the snapshot fails integrity checks, consolidation is blocked. If post-consolidation health checks fail, the system rolls back to the checkpoint automatically. No API exists to bypass this gate.
Design principle: The kernel is the smallest possible set of guarantees that cannot be violated. Everything else — intelligence, capabilities, strategies, UI — lives above the kernel in layers the agent can evolve. Safety comes from architecture, not from trust.

Extensibility: Agent Identity Primitives

Emerging patterns — agent-owned email addresses (agentmail), on-chain wallets, and machine-to-machine payment protocols (x402) — are gaining traction. Fawx does not implement these today, but the four-layer architecture accommodates them without kernel changes:

Wallets New AuthMethod::Wallet { chain, address } credential. Signing keys stored encrypted alongside API keys. The critical invariant holds: user-only permissions. An agent with a wallet still cannot authorize spending beyond what the user's policy allows. The kernel enforces the envelope; the loadable layer provides the capabilities.

Agentic Loop Specification

The execution cycle that drives every agent action — from perception through verification to memory

Loop Overview

Every agent action follows the same seven-step cycle. The kernel orchestrates the loop; loadable strategies influence decisions within it; the platform shell provides perception and executes physical actions.

State Machine

Idle → Perceiving → Reasoning → Deciding → Executing → Verifying → Learning → Continue / Complete

The Seven Steps

1

Perceive

Platform Shell Kernel Storage

What the agent knows right now. The shell provides user input and context. The kernel assembles it into a coherent context snapshot. Memory retrieval (loadable strategy) enriches the snapshot with relevant past context.

Inputs: User message, conversation history, any perception data from the shell (screen state, notifications, etc.)

Memory interaction: Retrieve relevant episodic + semantic memories. Working memory (conversation context) is already present. Retrieval strategy is loadable — the kernel calls it but doesn't dictate how it searches.

Output: PerceptionSnapshot — structured bundle of everything the agent knows at this moment

2

Reason

LLM Inference Loadable

What should the agent do? The perception snapshot is formatted into a prompt (loadable prompt template + strategy), sent to the LLM (model selection is loadable config), and the response is parsed into typed intents.

Inputs: PerceptionSnapshot, active prompt template, model routing config

Process:

1. Prompt construction — loadable strategy assembles system prompt + context + user message
2. Model selection — loadable config picks provider and model
3. LLM inference — kernel manages the API call, timeout, retry
4. Response parsing — extract tool calls as typed Intent[]

Output: Intent[] — typed action intents with expected evidence requirements. If no intents (pure text reply), skip to step 7 (Complete).

3

Decide

Kernel — Policy Engine Kernel — Budget Enforcer Kernel — Permission Registry

Is the agent allowed to do this? Every intent passes through three kernel-enforced gates. No loadable code can bypass these.

Gate 1 — Policy: Does this intent match an allow/deny/confirm rule? If deny → reject intent, return to Reason with constraint. If confirm → surface to user via platform shell, wait for approval.

Gate 2 — Budget: Can the agent afford this action? Check token budget, API cost estimate, task cost accumulator. If over budget → reject with budget exhaustion reason.

Gate 3 — Permission: Does the intent modify a protected layer? Check permission registry. If the layer is locked or requires approval → gate accordingly.

Output: Decision per intent — Approved, Denied(reason), or PendingConfirmation

On Denial

Denied intents are fed back to the Reason step with the denial reason as context. The agent gets a chance to reformulate. This is not a retry — it's a new reasoning step with additional constraints. Max reformulation depth is kernel-configured (default: 2).

4

Act

WASM Skill Platform Shell Kernel — State Machine

Execute the approved intent. The kernel dispatches the intent to the appropriate skill (WASM module) or platform capability. The state machine transitions to Executing.

Process:

1. Kernel resolves intent → skill (tool registry lookup)
2. Kernel checkpoints state (pre-action) for crash recovery
3. Skill executes — may call platform shell (e.g., tap element, read screen)
4. Skill returns ActionResult with evidence artifacts
5. Platform shell provides post-action state (screen content, etc.)

Inputs: Approved Intent, pre-action state

Output: ActionResult — success/failure indicator + evidence artifacts (screenshots, element state, page content, etc.)

5

Verify

Kernel — Verification Engine Kernel — State Machine

Did it actually work? The tool said "success" — prove it. The verification engine checks evidence artifacts against the intent's expected outcomes. This is kernel-enforced, not optional.

Checks:

• Artifact contracts — does the evidence match what "done" looks like for this intent type?
• Screen state diff — did the screen actually change in the expected way?
• Side-effect guards — did the action claim success without evidence of the side effect?

Outcomes:

Verified → state transitions to Learning
Failed → recovery strategy consulted (loadable), may retry or escalate
Inconclusive → additional evidence gathering (e.g., take screenshot, re-read screen)

Retry / Recovery

On failure, the kernel consults the active recovery strategy (loadable). The strategy may recommend: retry same action, try alternative approach, escalate to user, or abort. Retry budget is kernel-enforced (max retries per intent, max consecutive failures across intents). Recovery strategies can suggest, but cannot override the budget.

6

Learn

Kernel — Memory Policy Memory Strategies Storage Layer

What should the agent remember? After each action, the agent updates its memory systems. What gets stored is gated by kernel-enforced memory policies (user-controlled). How it's stored and indexed is determined by loadable memory strategies.

Updates by memory type:

Working memory: Append action result to conversation context. Run compaction if context window is approaching limits (compaction strategy is loadable).

Episodic memory: Record what happened — intent, action, outcome, timestamp. Gated by retention policy (user-controlled: "don't remember browsing history").

Semantic memory: Extract durable facts if applicable ("user's dentist is Dr. Smith on Oak St"). Gated by privacy policy. Indexing strategy is loadable.

Procedural memory: If a novel strategy worked (or a standard strategy failed repeatedly), propose updating the strategy. This goes through the A/B slot lifecycle with permission gating.

7

Continue or Complete

Kernel — State Machine

Are there more intents? The kernel evaluates whether the task is complete.

Continue: More intents from the same reasoning step → back to step 3 (Decide) for next intent. All intents processed but task not done → back to step 1 (Perceive) for fresh context + new reasoning.

Complete: Goal achieved (verified), max steps reached, user cancelled, or unrecoverable failure. Checkpoint final state, respond to user.

Completion gate: The kernel checks whether the final text response actually addresses the user's request (completion contract). A response that says "I opened the app" when the user asked "send a message to Chris" is not complete — the loop continues.

↺ Loop back: Step 7 → Step 1 (fresh perception for next action cycle) or Step 7 → Step 3 (more intents from same reasoning)

Interrupt Handling

The loop must handle external events at any point — user messages, cancellation, connection loss, budget exhaustion. The kernel checks for interrupts at defined boundary points.

User Steer (new message mid-task)

Checked at: pre-batch boundary, post-action boundary

User sent a new message while the agent is working. The message is delivered to conversation history. The current action completes, then the loop re-enters Perceive with the new context.

Action: Complete current action → deliver steer → re-Perceive

User Cancel

Checked at: every boundary point + mid-action for composite actions

User pressed Stop. The kernel halts execution at the nearest atomic boundary and assesses rollback.

Atomic actions (single tap, keystroke, API call): Already in-flight at the OS level — let it land, but do not proceed to the next step. No rollback needed.
Composite actions (multi-step: typing a paragraph, filling a form): Stop at the current atomic boundary. Roll back to pre-action checkpoint if partial state is worse than pre-action state.
Side-effect actions (message sent, form submitted, purchase made): If the side effect already fired, it cannot be recalled — log what completed and inform the user. If not yet fired, cancel prevents it.

After halt: restore from pre-action checkpoint (unless the action completed atomically and cleanly). Exit with cancelled reason. Checkpoint final state for potential resume.

Action: Halt at atomic boundary → assess rollback → restore checkpoint if partial → inform user what completed → exit

Connection Lost

Checked at: post-action boundary

The shell disconnected or the LLM provider connection dropped. The agent cannot proceed with the current cycle. Loop exits with connection_lost.

Action: Exit loop → notify user → wait for reconnection

Budget Exhausted

Checked at: Decide step (Gate 2)

The task has exceeded its cost budget. No more API calls allowed. The agent responds with what it has so far and explains the budget limit.

Action: Exit loop with partial result → explain budget limit

Each loop step interacts with memory differently. The kernel enforces what CAN be stored; loadable strategies determine HOW.

Working Memory

📖 Read at: every step (it IS the conversation context)
✏️ Written at: Step 4 (action results appended), Step 6 (compaction)

In-kernel, ephemeral. Compaction strategy (loadable) decides what to keep vs summarize vs drop when approaching context limits.

User Identity Model

The agent serves one human. It learns who they are — not as a profile to optimize against, but as a person to understand. Identity shapes every perception, every plan, every action, and every judgment of success.

Core principle: A generic agent that completes tasks is a tool. An agent that completes tasks the way its human would want them completed — anticipating preferences, matching communication style, respecting boundaries it was never explicitly told about — is a partner. That's what we're building.

Identity Dimensions

The agent builds a multi-dimensional understanding of its human. These aren't static profile fields — they're living representations that evolve with every interaction.

Core Identity

Who the person is. Name, timezone, language, occupation, life context. The foundation that rarely changes but informs everything.

"Joe. Mountain Time. Night owl. Software engineer. Builds in his free time."
Source: explicit (user tells agent) + inferred (usage patterns)

Communication Style

How the person talks and wants to be talked to. Formality level, verbosity preference, humor tolerance, emoji usage, response length expectations. Whether they want explanations or just results.

"Concise. No filler. Okay with casual tone. Prefers results over process narration. Uses direct questions."
Source: inferred from conversation patterns + explicit corrections

Taste & Preferences

What the person likes, dislikes, and is indifferent to. Restaurants, music, brands, UI aesthetics, tool choices, workflow preferences. Not just what they choose — why they choose it.

"Prefers dark mode. Orders Thai frequently. Chooses window seats. Likes minimal UI."
Source: observed choices + explicit statements + dreaming consolidation

Routines & Rhythms

Temporal patterns. When they wake, work, eat, exercise, wind down. Weekly rhythms. Seasonal patterns. What they do during commutes. When they're most/least receptive to interruptions.

"Doesn't like meetings before noon. Most productive 2-6pm. Checks email first thing but doesn't reply until after coffee."
Source: sensor data + calendar + interaction timing patterns

Social Graph & Context

The people in their life and the relationships. Not just contact names — the dynamics. Who they text daily vs yearly. Formal vs informal relationships. Who makes them happy.

"Texts mom every Sunday. Professional tone with boss. Casual shorthand with best friend. Mentions partner frequently."
Source: communication patterns + explicit mentions + calendar context

Values & Boundaries

What matters to them beyond tasks. Privacy sensitivity. Risk tolerance. Ethical preferences. Topics they avoid. Domains where they want control vs domains where they want autonomy.

"Privacy-conscious with health data. Risk-tolerant with investments. Wants full control over social media posts. Fine with autonomous scheduling."
Source: explicit policy choices + inferred from corrections + dreaming patterns

Identity in Every Loop Step

Identity isn't a separate system — it's woven into every step of the agentic loop. Here's how each step changes when the agent knows its human.

1

Perceive — Identity-Filtered Attention

The agent doesn't perceive everything equally. Identity tells it what matters to this person right now. A notification from their partner is high-priority. A promotional email is noise. A calendar reminder at 8am matters because they're a night owl who might oversleep.

Identity inputs: Routines (is this unusual timing?), Social graph (who is this from?), Values (is this a sensitive domain?)
Without identity: All notifications are equal. All screen content is equal priority. No personalized salience.
2

Reason — Identity-Informed Planning

The agent plans in a way that matches the person's preferences and style. "Book a flight" isn't just "find the cheapest flight." It's "find a window seat on United (their preferred airline), afternoon departure (they're a night owl), with enough layover time (they don't like rushing)."

Identity inputs: Taste (airline, seat, timing preferences), Communication style (how to phrase confirmations), Routines (time constraints), Values (budget sensitivity)
Without identity: Books the first available option. Generic confirmation phrasing. No preference application.
3

Decide — Identity-Aware Policy

Policies can be identity-conditioned. The same action might be auto-approved for one domain and require confirmation for another — based on the person's values and trust levels. "Send a message to partner" → auto-approve. "Post on social media" → always confirm.

Identity inputs: Values (control preferences per domain), Social graph (trust level per contact), Boundaries (never-auto-approve categories)
Without identity: Uniform policy regardless of context. Same confirmation threshold for everything.
4

Act — Identity-Styled Execution

When the agent composes a message, it writes in the person's voice — their vocabulary, their level of formality, their emoji usage. When it fills out a form, it uses their preferred defaults. When it navigates an app, it follows their habitual path (settings → advanced, not settings → basic).

Identity inputs: Communication style (tone, formality, vocabulary), Routines (habitual navigation paths), Taste (default choices for forms/selections)
Without identity: Generic tone. No vocabulary matching. Default form values. Standard navigation paths.
5

Verify — Identity-Calibrated Success

"Did it work?" is different for every person. Finding a restaurant isn't verified by returning results — it's verified by returning results this person would actually go to. The agent knows that success means matching the person's taste, not just completing the mechanical task.

Identity inputs: Taste (does this result match their preferences?), Values (does this respect their constraints?), Quality bar (how good is "good enough" for this person in this domain?)
Without identity: Mechanical completion check. Restaurant found = success. No taste calibration.
6

Learn — Identity Evolution

Every interaction is an identity signal. The user corrected the tone → communication style update. They rejected the restaurant → taste refinement. They overrode the schedule → routine adjustment. The identity model evolves continuously through the Learn step.

Identity inputs: User corrections (explicit signal), Choice patterns (implicit signal), Reaction timing (hesitation = uncertainty), Overrides (strong signal of misalignment)
Without identity: Task outcome logged. No personal learning. Same mistakes repeated.
7

Continue/Complete — Identity-Aware Completion

Completion criteria are identity-dependent. A perfectionist wants exhaustive results before the task is "done." A pragmatist wants the first good-enough option. The agent knows which one its human is, and for which domains.

Identity inputs: Quality bar per domain, Patience/thoroughness preference, Whether this person wants a summary or just the result
Without identity: Generic completion heuristic. Same thoroughness for everyone.

How Identity Learns

The identity model is built from five signal types, ranging from strong explicit signals to subtle implicit patterns.

Signal Strength Example Updates which dimension
Explicit statement 🟢 Strongest "I prefer afternoon flights" Taste, Routines
Explicit correction 🟢 Strong "Too formal — talk to me like a friend" Communication Style

Profile Evolution Over Time

The identity model isn't built at setup — it grows. The agent starts knowing almost nothing and gradually develops a deep understanding.

Day 1 — Blank Slate

Name, timezone, language. Everything else is default. The agent asks more than it infers. Confirmations are frequent. Responses are generic but careful. "I'm still learning your preferences — should I book the first available flight, or do you have a preferred airline?"

Week 1 — Basic Preferences

Communication style emerging (formality level, verbosity). A few taste signals (food, apps, scheduling patterns). First routine patterns detected. Agent still confirms frequently but starts offering informed suggestions. "I noticed you prefer afternoon slots — want me to filter for those?"

Month 1 — Working Model

Solid communication style match. Core taste profile established. Routines understood (morning/evening patterns, weekly rhythms). Social graph basics (frequent contacts, relationship tones). Agent confirms less, suggests more, gets it right most of the time.

Month 6 — Deep Understanding

Nuanced taste (not just "likes Thai" but "prefers Thai Palace on weeknights, Siam Orchid for dates"). Communication style adapts per context (casual with friends, professional in work mode). Anticipation loop starts being useful — agent knows what to offer before being asked.

Year 1+ — Partner

The agent knows things the person hasn't explicitly said but would agree with. It makes judgment calls the person would make. It notices when something is off ("you usually text your mom on Sundays — want me to remind you?"). The identity model is now a genuine understanding, not a preference database.

Privacy & Trust

Identity Data is the Most Sensitive Data

The identity profile is a detailed model of a person. It must be treated with extreme care.

🔒 Never leaves the device. The identity profile is stored in encrypted local storage. It is never sent to cloud providers, analytics services, or any external endpoint. LLM prompts receive identity-derived instructions ("use casual tone"), never raw profile data.
🔒 User can view everything. The full identity profile is viewable in settings. Every dimension, every data point, every confidence score. No hidden modeling.
🔒 User can edit or delete any field. "That's not right" → immediate correction. "Don't track this" → dimension disabled. "Forget everything about my food preferences" → taste dimension wiped.
🔒 Category opt-outs are permanent. If the user says "never model my health data," the kernel enforces this. No loadable strategy can override it. The prohibition is stored in the permission registry, not in the profile itself.
🔒 Social graph is derived, not scraped. The agent builds social context from interactions it's part of — not by crawling contacts, reading all messages, or analyzing photos. It knows about people the user has mentioned or communicated with through the agent.
The trust equation: The agent's usefulness grows with identity depth. But identity depth requires trust. And trust requires transparency and control. The system must prove it's trustworthy before the human will let it learn deeply. This is why the profile is fully visible, fully editable, and bounded by hard privacy rules from day one. The agent earns depth by deserving it.

Memory Systems

Four memory types with distinct lifecycles, storage models, and permission boundaries. Memory feeds the loop. The loop feeds memory. Together they make the agent learn.

Core principle: Memory is not a database — it's a learning system. Each memory type serves a different cognitive function, has a different lifecycle, and is controlled differently. The separation is architectural, not cosmetic. Mixing them would compromise both safety and intelligence.

The Four Memory Types

Working Memory

Kernel-managed

What it holds: Current conversation context, screen state, active task state, pending intents, action results in flight. Everything the agent needs right now to continue its current task.

Lifecycle: Ephemeral — created when a task starts, dies when the task completes (or is abandoned). Never persisted to storage. Never checkpointed for dreaming.

Size management: Bounded by the context window. When approaching limits, the kernel invokes a compaction strategy (loadable) that summarizes older entries, drops low-relevance content, and preserves high-signal items. The kernel enforces the size limit; the strategy decides what to compact.

Analogy: A whiteboard in front of you during a meeting. When you run out of space, you erase the least important stuff. When the meeting ends, the board is wiped.

Episodic Memory

Storage layer Kernel-gated writes

What it holds: What happened, when, and what the outcome was. Task records: intent → action → result → verification outcome → timestamp. Interaction history: what the user asked, what the agent did, whether it worked.

Lifecycle: Persistent with time-decay. Every episode has an importance score that decays over time unless reinforced by retrieval or outcome contribution. Entries below the importance threshold are candidates for pruning during dreaming cycles.

Storage: Encrypted, in the storage layer. Each entry includes: content, timestamp, importance score, embedding vector, provenance metadata, category tags.

Retrieval: During the Perceive step, the retrieval strategy (loadable) searches episodic memory for entries relevant to the current context. Importance-weighted: higher-scoring entries get retrieval priority within the context budget.

Analogy: Your diary. You remember what happened each day, but older entries fade unless they were important or you revisited them.

Semantic Memory

Storage layer Kernel-gated writes

What it holds: Durable facts, preferences, knowledge. Things that are true independent of when they were learned. "Joe's timezone is Mountain." "Thai Palace is on 5th Street." "The dentist appointment is with Dr. Smith." Also: the identity profile (see User Identity Model).

Lifecycle: Persistent, curated. Not time-decaying by default (facts don't expire just because they're old). Can be updated when new information contradicts old facts. Can be deleted by the user at any time.

Storage: Encrypted, indexed for efficient retrieval. Each entry includes: content, confidence score, embedding vector, provenance chain (which episodic memories led to this fact), category, last-verified timestamp.

Retrieval: During Perceive, semantic entries relevant to the current context are loaded. The identity profile is always loaded (kernel-enforced). Other entries use the same importance-weighted retrieval as episodic, but without time-decay bias.

Promotion from episodic: During dreaming, consolidation detects patterns in episodic memory and promotes them to semantic facts. "Joe ordered from Thai Palace 4 times this month" (episodic pattern) → "Joe's frequent Thai restaurant is Thai Palace" (semantic fact). Promotion requires a provenance chain — every semantic fact must trace back to its source episodes.

Analogy: Your knowledge base. You know facts without remembering exactly when you learned them, but you trust them because they've been reinforced by experience.

Procedural Memory

Loadable layer (A/B slotted) Kernel-gated modifications

What it holds: How to do things. Reasoning strategies, recovery strategies, prompt templates, compaction strategies, memory retrieval strategies, tool preferences, navigation patterns. The agent's learned behaviors, not its learned facts.

Lifecycle: Persistent, versionable. Changes go through the A/B slot lifecycle — proposed by the agent (or by the system during dreaming), validated by the kernel, gated by the permission registry, deployed with automatic rollback on failure.

Storage: In the loadable layer, not the storage layer. Procedural memory is the loadable configuration — strategies, templates, and configs that shape how the loop executes.

Learning: During the Learn step, if a strategy consistently underperforms (repeated verification failures, high retry rates), the agent may propose an updated strategy. During dreaming, consolidated outcome patterns can trigger strategy evolution proposals.

Analogy: Muscle memory + habits. You don't think about how to ride a bike — you just do it. But the way you ride was learned through practice, and it can be updated if you learn a better technique.

Retrieval Architecture

Memory retrieval happens at the Perceive step. The kernel calls the retrieval strategy (loadable); the strategy returns ranked memory hits within a token budget.

Retrieval Pipeline

Current context → Embed query → Candidate search → Importance-weighted rank → Token budget fit → MemoryHits[]
Pipeline stageOwnerDetails
Embed query Loadable Convert current context (user message + task state) into an embedding vector. Strategy chooses: full context embedding, keyword extraction, or hybrid.
Candidate search Storage Vector similarity search across episodic + semantic stores. Returns top-K candidates by cosine similarity. K is configurable (default: 50).
Importance-weighted rank Loadable Re-rank candidates using composite importance score. Balances relevance (embedding similarity) with importance (retrieval frequency, outcome contribution, recency). Weighting function is loadable.
Token budget fit Kernel Pack ranked results into the memory portion of the context window. Budget is kernel-determined based on total context limits minus conversation + system prompt. Truncates at the budget boundary.

Special Retrieval Rules (Kernel-Enforced)

🔒 Identity profile is always loaded. The user identity profile (semantic memory) is loaded into every PerceptionSnapshot regardless of relevance scoring. The agent never reasons without knowing who it serves.
🔒 Retrieval is logged. Every retrieval (which entries were returned, which were considered but excluded) is recorded. This data feeds importance score updates: retrieved entries get their retrieval_freq incremented.

Write Gating

Writing to persistent memory (episodic, semantic) is gated by kernel-enforced policies. The agent proposes writes during the Learn step; the kernel checks each against the user's memory policy before committing.

GateChecksOn failure
Category filter Is this memory in an allowed category? User can block entire categories ("never store health data", "don't remember browsing history"). Write silently dropped. Logged in audit trail.
Privacy filter Does this memory contain PII or sensitive data that the user hasn't consented to storing? Checked against the privacy policy in the permission registry. Write blocked. Logged. Agent informed (can try reformulating without sensitive data).
Deduplication Is this substantially similar to an existing entry? Vector similarity check against recent entries to prevent redundant storage. Existing entry's importance score updated (reinforcement). New entry discarded.
Provenance requirement Does the write have a valid provenance chain? Semantic entries must trace to source episodes. Fabricated memories are rejected. Write rejected. Logged as anomaly.

Graph Index (Design Now, Build Later)

Semantic memory entries are flat by default — connected only by embedding similarity. But real-world knowledge is relational: people know people, places contain things, events involve participants. A graph index over semantic memory captures these relationships and enables traversal, not just search.

Architecture decision: The graph is an index structure over semantic memory, not a fifth memory type. The four memory types stay the same. The graph is a complementary retrieval path — vector similarity for "what's relevant?" and graph traversal for "what's connected?"

How the Graph Gets Built

Entity extraction Learn step (every loop cycle) When episodic entries are created, extract named entities (NER). Tag the episode with entity refs. Lightweight — no LLM needed, rule-based or small local model.

Graph-Enhanced Retrieval

During Perceive, the retrieval pipeline gains a second path:

Context → Extract entities → Graph neighborhood → Merge with vector results → Enriched MemoryHits[]

Vector path (existing): embed context → similarity search → importance-weighted rank → results.
Graph path (new): extract entities from context → traverse 1-2 hops in graph → return connected entities + their semantic entries → merge with vector results, dedup, re-rank within token budget.

The graph path catches things vector similarity misses. "Message from Dr. Smith" → graph expands to: Dr. Smith works at Oak Street Dental, appointment is next Tuesday, last visit was January. Embedding similarity alone might surface "Dr. Smith" entries but miss the connected context.

What This Enables

CapabilityWithout graphWith graph
Social graph Flat semantic entries: "Mom's name is Sarah." No relationship structure. Full relationship model: Sarah → mother_of → Joe, Sarah → lives_in → Denver, Joe → texts → Sarah (weekly, Sundays).
Multi-hop reasoning "Where does Joe's dentist work?" requires the right entry to be retrieved by embedding similarity. Traverse: Joe → visits → Dr. Smith → works_at → Oak Street Dental. Answer found by structure, not search luck.
Context enrichment Retrieved memories are independently relevant entries. Retrieved memories include relationship context: not just "Dr. Smith" but the full neighborhood of connected facts.
Dreaming quality Pattern detection via embedding clustering. Pattern detection via graph structure: "these 4 episodes all involve the same entity cluster" is a stronger signal than embedding proximity.

Memory Transitions

Memory doesn't just sit in stores — it moves between them. Each transition has explicit criteria, quality gates, and a kernel/loadable split. Getting these transitions right is what makes the difference between an agent that accumulates data and one that actually learns.

Transition 1: Working → Episodic (at the Learn step)

When a task ends, working memory dies. The Learn step extracts what's worth keeping before that happens. This is a classification problem — most of working memory is transient context that has no long-term value.

✓ Should become an episode

• Any action that went through Decide→Act→Verify (the agent did something — record intent, action, outcome)
• User statements with semantic content ("I prefer window seats", "my dentist is Dr. Smith")
• User corrections and overrides (always — strongest learning signal)
• Novel information that contradicts or extends existing semantic memory
• Notable failures (learning from mistakes > recording successes)
• Task completion outcomes (goal achieved? partially? what was the gap?)

✗ Should NOT become an episode

• Routine acknowledgments ("ok", "thanks", "got it")
• Information already in semantic memory with high confidence (reinforce existing entry via dedup, don't create new episode)
• Intermediate navigation steps (opened app, tapped button) unless they failed or were novel
• Working memory compaction artifacts (summaries of older context)
• Repeated re-reads of the same screen content
• System-level noise (retry attempts, timeout recoveries)

Transition 2: Episodic → Semantic (during dreaming Phase 2)

This is where patterns become facts. The consolidation engine detects recurring themes across episodes and promotes them to durable semantic entries. Quality matters more than quantity — a bad semantic fact (wrong preference, incorrect relationship) actively degrades the agent's behavior.

Promotion Criteria

CriterionThresholdRationale
Frequency Pattern appears across ≥3 episodes (configurable) One Thai Palace order isn't a pattern. Four is. Prevents coincidences from becoming facts.

Importance Source episodes have above-median composite importance Patterns from important episodes (frequently retrieved, contributed to successes) are more likely to be real signals.
Novelty No existing semantic fact covers this pattern If it's already a semantic fact, reinforce the existing entry (boost confidence, extend provenance) rather than duplicating.

Quality Gates (preventing bad promotions)

🔒 Provenance chain required. Every semantic fact must trace to its source episodes. The kernel rejects promotions without provenance. No facts without evidence.
🔒 Contradiction detection. If a proposed fact contradicts an existing semantic entry, the kernel does not blindly overwrite. It compares evidence strength (episode count, importance scores, recency). The better-supported fact wins. The loser gets confidence reduced, not deleted. The contradiction is logged in the audit trail.
🔒 Confidence calibration. New semantic facts from inference start at moderate confidence (~0.6), never high. They earn high confidence through reinforcement by future episodes. This prevents a single dream cycle from creating "facts" that are actually coincidences.
🔒 Explicit always beats inferred. A user's explicit statement ("I'm vegetarian") immediately creates a high-confidence semantic fact (0.95). Inferred patterns always start moderate (0.6). One explicit correction outweighs 50 inferred episodes. The kernel enforces this asymmetry.

Transition 3: Episodic → Procedural (behavioral learning)

The subtlest transition — when does the agent learn a new behavior, not just a new fact? Procedural updates change how the agent acts, not what it knows. They carry higher risk (a bad strategy degrades all future tasks of that type) and go through the full A/B slot lifecycle.

Transition Summary

The principle: Each transition has ascending stakes. Working→Episodic is cheap (recording what happened — low risk, high volume). Episodic→Semantic is moderate (distilling patterns into facts — medium risk, requires quality gates). Episodic→Procedural is expensive (changing behavior — high risk, requires A/B validation). The quality gates and approval requirements scale with the stakes. The kernel enforces this escalation — loadable strategies cannot skip gates or bypass the A/B lifecycle.

Memory Consolidation ("Dreaming")

How the agent strengthens what matters, prunes what doesn't, and distills experience into knowledge — all while keeping a safety net that guarantees nothing is lost without the user's consent.

Why "dreaming": Human memory consolidation happens primarily during sleep — the brain replays experiences, strengthens important connections, and prunes noise. Fawx does the same during idle time. The agent doesn't just accumulate data; it processes experience into intelligence. But unlike human sleep, every consolidation cycle is checkpointed and reversible.

When Dreaming Happens

The dream cycle is triggered by the kernel, not by the agent. The agent cannot initiate or skip consolidation.

TriggerConditionPriority
Scheduled User-configured schedule (default: nightly when charging, between 2-5 AM local time). Requires device to be idle (no active tasks) and charging. Normal — full consolidation cycle
Memory pressure Episodic memory store exceeds the soft size limit (default: 80% of allocated storage). Triggers early pruning pass to free space. Elevated — pruning-only (no promotion or strategy evolution)
Extended idle Device has been idle for >30 minutes during waking hours. Lightweight consolidation pass — scoring and importance decay only, no LLM inference. Low — scoring-only
User-initiated User explicitly triggers consolidation from settings ("Process my memories now"). Full cycle. Normal — full consolidation cycle
🔒 Dream cycles never interrupt active tasks. If the user starts interacting during a dream cycle, consolidation is immediately paused and will resume at the next trigger. In-progress mutations are rolled back to the pre-dream checkpoint.

The Six Consolidation Phases

A full dream cycle runs six phases in order. Each phase is independently checkpointed — if interruption occurs, only the current phase is rolled back, not the entire cycle.

Phase 2 — Consolidate (Episodic → Semantic Promotion)

Consolidation strategy: Loadable Provenance enforcement: Kernel

Purpose: Detect patterns across episodic memories and promote them to semantic facts.

Process:

1. Cluster detection — The consolidation strategy (loadable) identifies clusters of related episodic entries. Clustering is by embedding similarity + category + temporal proximity.
2. Pattern extraction — For high-importance clusters with ≥3 entries, the strategy proposes a semantic fact. This requires LLM inference: the cluster content is summarized into a durable fact statement.
3. Provenance chain — The kernel verifies that every proposed semantic fact has a valid provenance chain to its source episodes. No fact without evidence.
4. Deduplication check — If a similar semantic fact already exists, the existing fact is reinforced (confidence score boosted, provenance chain extended) rather than creating a duplicate.
5. Write gate — New semantic facts pass through the standard memory write gates (category filter, privacy filter, provenance requirement).

// Episodic cluster:
//   "Ordered pad thai from Thai Palace on Jan 3"
//   "Ordered green curry from Thai Palace on Jan 15"
//   "Ordered pad thai from Thai Palace on Jan 22"
//   "Ordered from Thai Palace on Feb 1"
//
// → Semantic fact:
//   "Joe's go-to Thai restaurant is Thai Palace. Favors pad thai."
//   Confidence: 0.82
//   Provenance: [episode_a, episode_b, episode_c, episode_d]

Cost note: Pattern extraction requires LLM inference. The consolidation strategy controls batch size (how many clusters to process per cycle) and can throttle based on power/cost budgets. Only clusters above a minimum importance threshold trigger LLM calls.

Phase 3 — Strengthen

Decay rate adjustment: Kernel

Purpose: Reduce the decay rate on high-importance memories so they persist longer.

Entries with composite importance above the strengthening threshold (default: 75th percentile) get their decay curve flattened. This means they'll score higher for longer, making them more likely to survive future pruning cycles.

Strengthening is the mechanism by which frequently-useful memories become effectively permanent without manual pinning. A memory that keeps getting retrieved and contributing to successful outcomes will naturally resist decay.

Ceiling: Strengthened decay rate never reaches zero (only user-pinned memories have zero decay). Even highly important memories will eventually be pruned if they stop being useful — unless the user explicitly pins them.

Phase 4 — Prune

Pruning execution: Kernel Pruning aggressiveness: Loadable (bounded by user policy)

Purpose: Remove entries below the importance threshold to keep memory stores lean and retrieval quality high.

Pruning rules:

• Entries with composite importance below the pruning threshold are candidates
• User-pinned entries are never prunable (kernel-enforced)
• Entries with active provenance chains (they're the source for a semantic fact) are protected until the semantic fact has been independently reinforced ≥N times
• Entries in user-protected categories are subject to category-specific retention rules
• Maximum prune rate per cycle is capped (default: 30% of total entries) — prevents runaway pruning

Pruning is soft-delete first. Pruned entries are moved to a "pruned" partition, not immediately destroyed. They remain in the pruned partition for a grace period (default: 7 days / 1 dream cycle). After the grace period, if no rollback has been triggered, they're permanently deleted.

Phase 5 — Identity Consolidation

Identity strategy: Loadable Privacy enforcement: Kernel

Purpose: Update the identity profile based on accumulated interaction signals since the last dream cycle.

The identity consolidation strategy (loadable) processes queued identity signals — explicit corrections, choice patterns, interaction timing, overrides — and proposes identity profile updates. See User Identity Model — How Identity Learns for signal types and weighting.

Anomaly guard: If any identity dimension's confidence score would change by more than the anomaly threshold (default: 0.3) in a single cycle, the update is flagged and the kernel runs the post-dream health check with elevated scrutiny on that dimension. Rapid identity drift usually indicates either a major life change (legitimate) or a consolidation bug (dangerous).

Phase 6 — Audit & Report

Audit: Kernel (append-only)

Purpose: Log everything that happened during consolidation and generate an optional dream report for the user.

Audit log entries (always written):

• Number of entries scored, consolidated, strengthened, pruned
• New semantic facts created (with provenance chains)
• Identity dimension changes (before/after confidence scores)
• Anomalies detected (if any)
• Checkpoint ID used, cycle duration, LLM inference cost

Dream report (on by default, user-configurable):

A human-readable summary of what the agent learned, what it remembered, and what it let go. Not pushed as a notification — accessible in a dedicated, easy-to-find location in the app (e.g., Memory section in settings, or a "Last night" card on the home screen). Always available on demand, never streaming at the user.

Consolidation Strategy Evolution

The consolidation strategies themselves (scoring weights, clustering parameters, promotion thresholds) are procedural memory — they live in the loadable layer and are subject to A/B slot evolution.

How consolidation improves itself:

1. The kernel tracks consolidation quality metrics: pruned entries that were later searched for (miss), promoted facts that were never retrieved (waste), and entries that survived pruning and proved useful (hit).
2. If miss rate exceeds the threshold, the system proposes adjusting scoring weights (e.g., increase recency weight to keep more recent entries).
3. Strategy updates go through A/B slots: new weights are deployed to the pending slot, run for N cycles, compared against the fallback slot's historical metrics, and promoted or rolled back.

This means the agent gets better at dreaming over time — not just better at doing things, but better at remembering the right things.

Dreaming Invariants

🔒 Consolidation cannot begin without a validated checkpoint. Atomic snapshot of all persistent stores, integrity-verified, before any mutations. See Kernel Architecture.
🔒 Dream cycles never interrupt active tasks. If user interaction begins during consolidation, the cycle is paused and in-progress mutations are rolled back.
🔒 User-pinned memories are never prunable. No scoring threshold, consolidation strategy, or storage pressure can override a user pin.
🔒 Pruning is soft-delete with grace period. Pruned entries remain recoverable for the grace period before permanent deletion.
🔒 Semantic facts require provenance. No semantic entry can be created without a valid chain to its source episodic memories. The kernel rejects provenance-less promotions.
🔒 Maximum prune rate is capped. No single dream cycle can prune more than the configured maximum percentage of entries (default: 30%). This prevents runaway pruning from a bad strategy.
🔒 All consolidation actions are audited. Every creation, modification, strengthening, pruning, and identity update is logged to the append-only audit trail.

Memory-Augmented Intelligence

Four ascending layers of intelligence improvement — from context injection to genuine learning. Each layer makes the agent measurably better at serving its human. Each is achievable with increasing capability requirements.

The question: Can the agent's intelligence actually improve from experience, not just its knowledge? The answer is yes — through four distinct mechanisms, each more powerful than the last. The architecture supports all four from day one, even though the higher layers won't be active until later.

The Four Intelligence Layers

Layer 1 — Retrieval-Augmented Generation (RAG) BUILD NOW

PHASE 1

Mechanism: At the Perceive step, relevant memories are retrieved and injected into the LLM prompt. The model's weights don't change — but its effective behavior changes based on what context it receives.

How it improves over time: Importance-weighted retrieval (from the dreaming cycle) means frequently-useful memories naturally rise to the top of retrieval results. The agent gets better at surfacing the right context because the memory system learns which memories matter.

Layer 2 — Behavioral Conditioning via Procedural Memory BUILD NOW

Mechanism: Procedural memory (strategies, prompt templates, recovery patterns) directly conditions the agent's behavior. The LLM's weights don't change, but its instructions do — based on experience.

How it improves over time: Strategy A/B testing. When procedural memory proposes a new strategy (via A/B slots), the kernel can run both strategies for N tasks and compare success rates. The strategy with higher verified-success rate wins. This is evolution — not gradient descent, but it produces measurable behavioral improvement.

Layer 3 — Learned Retrieval Attention DESIGN NOW, BUILD LATER

Mechanism: Instead of fixed re-ranking functions, a small neural network learns to predict which memories will be useful for a given context. The network maps (context_embedding, memory_embedding) → relevance_score, trained on whether retrieved memories contributed to successful outcomes.

How it improves over time: The attention network is trained continuously on retrieval outcome data. Every time a retrieved memory contributes to a verified-successful task, that's a positive training signal. Every time a high-ranked memory is unused, that's a negative signal. The network learns what to remember when.

Safety Considerations

Fine-tuning can degrade model quality. Unlike Layers 1-3 (which add intelligence without modifying the model), Layer 4 modifies the model itself. Bad training data, overfitting, or catastrophic forgetting could make the model worse.

Mitigation:

• Keep the base model intact. Fine-tuning produces a LoRA adapter, not a modified base model. Rollback = remove the adapter.
• A/B evaluation before promoting. Run the fine-tuned model and base model side by side for N tasks, compare verified success rates.
• Pre-fine-tuning model checkpoint (analogous to pre-dream memory checkpoint). If the fine-tuned model underperforms, automatic rollback.
• User controls: "fine-tune my model" is an explicit opt-in, not a default behavior.

Anticipation Loop

Proactive background perception — the agent watches for conditions and offers help before the user asks. Transforms the agent from reactive tool to anticipatory partner.

The difference: A reactive agent waits for "book me a flight." An anticipatory agent notices you have a meeting in Denver tomorrow, you haven't booked a flight, and it's getting late — and offers to help. This is the line between a tool and a partner. But it must be done carefully: a proactive agent that guesses wrong is annoying. A proactive agent that acts without permission is dangerous.

Trigger Tiers

Not all proactive triggers are created equal. They're organized in three tiers by urgency and confidence requirements.

Tier 1 — Time-Critical Alerts

Confidence threshold: 0.9+ (very high — these interrupt)

Delivery: Immediate notification, even during light idle.

Examples:

• Meeting in 15 minutes, haven't left home yet, commute is 20 minutes
• Flight in 3 hours, haven't checked in, airline allows online check-in
• Unusual charge on credit card notification just arrived
• Birthday of close contact today, haven't sent a message yet

Signal sources: Calendar + location + time + notification content + social graph

Tier 2 — Contextual Suggestions

Confidence threshold: 0.75+ (high — these suggest, don't interrupt)

Delivery: Ambient UI element (badge, subtle card). Shown during natural interaction pauses.

Examples:

• At a restaurant the agent hasn't seen before → "Want me to save this to your places?"
• Weekly grocery order usually placed on Sundays, it's Sunday afternoon → "Want me to start your grocery order?"
• Approaching end of billing cycle, usage is high → "You've used 80% of your data plan"
• Repeated searches for a topic → "I found some related articles you might like"

Signal sources: Location + routines + episodic patterns + identity preferences

Tier 3 — Passive Insights

Confidence threshold: 0.6+ (moderate — these are offered, never pushed)

Delivery: Available on request or during daily summary. Never interrupts.

Examples:

• "You've been ordering from Thai Palace a lot — they have a loyalty program"
• "Your meeting pattern this week is heavier than usual — want me to block some focus time?"
• "You haven't backed up your photos in 2 weeks"
• "Based on your sleep patterns, you seem to get better rest when you stop using screens by 11pm"

Signal sources: Episodic patterns + semantic knowledge + routine analysis + dreaming consolidation insights

🔒 Proactive mode is suggest-only by default. The anticipation loop cannot execute actions autonomously. All proactive outputs are suggestions that require user confirmation. The user can relax this per-category, but the kernel default is always confirm-required.

Dangerous because unsolicited actions erode trust fast. The architecture ensures proactive behavior is always gated by high confidence, always suggestion-only by default, always fatigue-managed, and always disableable. The agent earns the right to anticipate by being right — and learns to stay quiet when it's not.

Expectation-Verification

The agent predicts what should happen before acting, then compares prediction to reality. Makes verification faster, catches subtle failures, and generates a powerful learning signal.

The insight: Current verification checks evidence against static contracts — "did a confirmation dialog appear?" But the agent has context the contract doesn't: it knows what app is open, what screen it's on, what the expected next screen should look like. A prediction generated before the action is a much richer verification target than a generic contract. When prediction matches reality, verification is instant. When it doesn't, the mismatch is itself a valuable signal.

How It Works

Modified Intent Structure

}

  confidence:     f32,                        // How confident the agent is in this prediction

Modified Verification Flow

Reason: generate intent + prediction → Act: execute → Verify: two-pass check 
fn verify(result: ActionResult, intent: Intent) → VerificationOutcome:
  // PASS 1: Artifact contract check (KERNEL-ENFORCED, HARD GATE)
  // This is the existing verification — does evidence meet the contract?
  contract_result = check_artifact_contracts(result, intent.evidence_requirements)

  if contract_result == Failed:
    return VerificationOutcome::Failed(contract_result.reason)

  // PASS 2: Prediction comparison (LOADABLE, SOFT SIGNAL)
  // Compare actual outcome to predicted outcome
  if let Some(prediction) = intent.expected_outcome:
    prediction_match = compare_prediction(prediction, result.post_action_state)

    match prediction_match:
      Match(score > 0.8) →
        // Prediction was accurate — strong confidence in success
        // Learning signal: boost this intent type's prediction accuracy
        record_prediction_accuracy(intent, PredictionAccurate)
        return VerificationOutcome::Verified(confidence: High)

      PartialMatch(score 0.4-0.8) →
        // Prediction partially matched — something unexpected happened
        // Don't fail, but flag the surprise for learning
        record_prediction_accuracy(intent, PredictionPartial, delta: mismatch_details)
        return VerificationOutcome::Verified(confidence: Medium, surprise: mismatch_details)

      Mismatch(score < 0.4) →
        // Prediction was wrong — the world looks different than expected
        // Contract passed but prediction failed. This is a surprise.
        // Escalate to evidence-gathering sub-loop
        record_prediction_accuracy(intent, PredictionMismatch, delta: mismatch_details)
        return VerificationOutcome::Inconclusive(reason: PredictionMismatch)

  // No prediction generated — fall back to contract-only result
  return VerificationOutcome::from_contract(contract_result)

The Two-Pass Architecture

Pass 1: Artifact Contract (Kernel)

Purpose: Hard safety gate. Does the evidence meet the minimum requirements for "done"?

Examples: Confirmation dialog appeared. File exists at expected path. HTTP 200 received. Element with expected text is present.

Properties: Deterministic. Static. Same contract for every "tap button" action regardless of context. Cannot be bypassed.

Failure mode: If Pass 1 fails, the action failed. Period. Prediction is not consulted.

Pass 2: Prediction Comparison (Loadable)

Purpose: Intelligent verification. Does the world look like the agent expected? Catches subtle failures that pass contracts but indicate something is wrong.

Examples: Expected to see the home screen after pressing back, but see a dialog instead. Expected the message to appear in the conversation, but the conversation is empty. Expected the search to return restaurant results, but got an error page.

Properties: Context-dependent. Different prediction for every action based on the agent's understanding of the current state. Soft signal — informs learning, doesn't hard-gate.

Failure mode: If Pass 2 mismatches, it doesn't mean the action failed — it means something unexpected happened. Could be: (a) the agent's model of the world was wrong, (b) a subtle failure the contract didn't catch, or (c) a benign UI variation. The mismatch triggers investigation, not automatic failure.

Why two passes instead of one: Safety and intelligence have different requirements. The artifact contract is a hard floor — it prevents the agent from falsely claiming success. The prediction comparison is a soft ceiling — it tells the agent "yes, you succeeded, but was it what you expected?" Combining them gives both reliability (never miss a real failure) and intelligence (catch subtle failures and learn from them).

Recursive Loop & Task Decomposition

The loop as a recursive function — not just for task decomposition, but for verification, navigation, recovery, and any situation where a step needs more intelligence than a single pass provides.

Why recursion: The seven-step loop is linear in specification but recursive in practice. During Reason, the agent might need to Perceive more. During Act, an action might require navigation. During Verify, inconclusive results need a fresh Perceive→Reason cycle. Task decomposition means sub-goals each run their own loop. The linear loop is a simplification that breaks on real-world tasks. Making recursion explicit is how we handle complexity without abandoning the safety model.

The Recursive Signature

Loop(goal: Goal, context: LoopContext, depth: u8) → LoopResult

where:
  Goal     = what this loop invocation is trying to achieve
  Context  = parent's working memory (read-only) + this level's working memory (read-write)
  depth    = current recursion depth (0 = top-level user request)

returns:
  LoopResult::Completed(evidence)      // Goal achieved, verified
  LoopResult::Failed(reason)           // Unrecoverable failure
  LoopResult::NeedsUser(context)       // Escalation — user input needed
  LoopResult::DepthExceeded            // Hit max recursion depth
  LoopResult::BudgetExhausted          // Hit budget limit for this sub-loop
  LoopResult::Interrupted(checkpoint)  // User cancelled or steered

Full Recursive Loop

Loop(goal, context, depth) → LoopResult:
  // KERNEL GATE: depth limit
  if depth > MAX_DEPTH:
    return LoopResult::DepthExceeded

  // Step 1: PERCEIVE
  perception = perceive(context)

  // Step 2: REASON
  plan = reason(perception, goal)

  // If pure text response (no actions needed):
  if plan.is_response_only():
    return LoopResult::Completed(plan.response)

  for sub_goal in plan.goals:
    // Step 3: DECIDE
    decision = decide(sub_goal)

    match decision:
      Denied(reason) →
        // Feed denial back, re-reason with constraint
        plan = reason(perception.with_constraint(reason), goal)
        continue  // try re-planned goals

      PendingConfirmation →
        // Surface to user, wait for approval
        approval = await_user_confirmation(sub_goal)
        if !approval: continue

      RequiresDecomposition →
        // RECURSE: sub-goal needs its own loop
        sub_result = Loop(
          sub_goal,
          context.child_scope(budget_fraction),
          depth + 1
        )
        learn(sub_result)
        if sub_result.is_failure():
          recovery = consult_recovery(sub_result)
          match recovery:
            Replan → break   // back to top of loop with failure context
            Escalate → return LoopResult::NeedsUser(context)
            Abort → return LoopResult::Failed(sub_result.reason)
        continue

      Approved(intent) →
        // Step 4: ACT
        checkpoint_state()
        result = act(intent)

        // Step 5: VERIFY
        outcome = verify(result, intent.expected_outcome)

        match outcome:
          Verified →
            // Step 6: LEARN
            learn(VerifiedOutcome { intent, result, outcome })

          Failed →
            recovery = consult_recovery(outcome)
            match recovery:
              Retry → continue  // retry this sub_goal
              GatherEvidence →
                // RECURSE: evidence-gathering sub-loop
                evidence = Loop(
                  Goal::Verify(intent),
                  context.child_scope(evidence_budget),
                  depth + 1
                )
                re_outcome = verify_with_evidence(result, evidence)
                learn(re_outcome)
              Replan → break  // back to reason with failure context
              Escalate → return LoopResult::NeedsUser(context)
              Abort → return LoopResult::Failed(reason)

          Inconclusive →
            // RECURSE: gather more evidence
            evidence = Loop(
              Goal::Verify(intent),
              context.child_scope(evidence_budget),
              depth + 1
            )
            re_outcome = verify_with_evidence(result, evidence)
            learn(re_outcome)

  // Step 7: CONTINUE or COMPLETE
  completion = evaluate_completion(goal, all_outcomes)
  match completion:
    Complete → return LoopResult::Completed(evidence)
    Continue → return Loop(goal, context.updated(), depth)  // re-perceive
    MaxSteps → return LoopResult::Completed(partial_results)

When Recursion Naturally Occurs

Task Decomposition

Trigger: The Reason step determines the goal is too complex for a single action and outputs a TaskPlan with sub-goals.

Example: "Book a trip to Denver" → [search flights, select and book flight, add to calendar, text itinerary to partner]

Recursion type: Each sub-goal gets its own Loop(sub_goal, ...) invocation. Sub-goals can depend on each other (book flight must complete before adding to calendar) or be independent (text itinerary can happen after calendar but doesn't depend on it).

Inconclusive Verification

Trigger: The Verify step can't determine if the action succeeded from available evidence.

Example: Agent tapped "Send" but can't tell if the message actually sent. → Sub-loop: take screenshot, read screen, look for confirmation indicator.

Recursion type: Evidence-gathering sub-loop. Goal is Verify(original_intent). Typically shallow (depth+1 only, no further decomposition). Uses a smaller budget allocation.

Navigation Prerequisites

Trigger: The intended action requires navigating to a different screen/app first.

Example: "Type in the search box" but search box isn't visible → Sub-loop: open app, navigate to search screen, find and tap search box.

Recursion type: Prerequisite sub-loop. The parent loop's Act step discovers it can't execute directly and spawns a navigation sub-loop. Common in mobile contexts where actions require multi-step navigation.

Recovery Escalation

Trigger: An action failed and the recovery strategy suggests an alternative approach that's complex enough to need its own planning.

Example: Tapping a button failed (element not found) → recovery: scroll to find it, try different selector, or navigate to the feature via a different path.

Recursion type: Recovery sub-loop. Similar to navigation but triggered by failure. The sub-loop's goal is to achieve the same end state through an alternative path.

Context Management Across Depth

PropertyBehaviorRationale
Context inheritance Child loops can read parent working memory. Writes are scoped to child only. Child needs to know what the parent was doing (e.g., what app is open) but shouldn't pollute parent's context with navigation details.
Result propagation Child LoopResult flows back to parent as evidence. Parent's Learn step incorporates child outcomes. The parent needs to know if the sub-goal succeeded, failed, or was escalated. But it doesn't need the full trace.
Budget partitioning Parent allocates a budget fraction to each child. Child cannot exceed its allocation. Unspent budget returns to parent. Prevents runaway sub-loops from consuming the entire task budget. Forces the agent to allocate resources thoughtfully.
Independent checkpointing Each depth level checkpoints independently. Crash at depth 2 → resume at depth 2 without replaying depth 0-1. Crash recovery proportional to the interrupted work, not the total task. Essential for long multi-step tasks.
Memory access All depths share the same episodic/semantic memory stores for retrieval. Writes from child loops are gated by the same policies. The agent's knowledge shouldn't be siloed by recursion depth. A fact learned at depth 2 is available at depth 0.

State Machine with Recursion

The kernel maintains a stack of state machines, one per recursion level. Each level follows the standard 7-step state machine independently.

LoopStack {

  max_depth: u8,
}

LoopFrame {
  goal:           Goal,
  state:          LoopState,   // Idle | Perceiving | Reasoning | Deciding | ...
  working_memory: WorkingMemoryFrame,
  checkpoint:     Option<Checkpoint>,
  budget:         Budget,
  parent_context: ReadOnlyRef<WorkingMemoryFrame>,  // read-only view of parent
}

// State transitions happen within a single frame.

Design principle: Recursion is not a feature added to the loop — it's a fundamental property of how intelligent agents solve real problems. By making it explicit in the architecture, we get managed depth (safety), budget partitioning (resource control), independent checkpointing (crash resilience), and scoped context (clean separation) — all enforced by the kernel, all impossible to bypass.
