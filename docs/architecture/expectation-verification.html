Expectation-Verification

The agent predicts what should happen before acting, then compares prediction to reality. Makes verification faster, catches subtle failures, and generates a powerful learning signal.

The insight: Current verification checks evidence against static contracts — "did a confirmation dialog appear?" But the agent has context the contract doesn't: it knows what app is open, what screen it's on, what the expected next screen should look like. A prediction generated before the action is a much richer verification target than a generic contract. When prediction matches reality, verification is instant. When it doesn't, the mismatch is itself a valuable signal.

How It Works

Modified Intent Structure

Intent {
  action:               Action,          // What to do
  params:               Params,          // How to do it
  evidence_requirements: EvidenceReqs,   // What "done" looks like (existing)
  expected_outcome:     Option<ExpectedOutcome>,  // NEW: what the agent predicts will happen
}

ExpectedOutcome {
  screen_state:   Option<ScreenPrediction>,  // What the screen should look like after
  app_state:      Option<AppPrediction>,     // What app state should change
  side_effects:   Vec<SideEffectPrediction>, // Expected side effects (message sent, file created)
  confidence:     f32,                        // How confident the agent is in this prediction
  format:         PredictionFormat,           // structured | natural_language | embedding
}

Modified Verification Flow

Reason: generate intent + prediction → Act: execute → Verify: two-pass check 
fn verify(result: ActionResult, intent: Intent) → VerificationOutcome:
  // PASS 1: Artifact contract check (KERNEL-ENFORCED, HARD GATE)
  // This is the existing verification — does evidence meet the contract?
  contract_result = check_artifact_contracts(result, intent.evidence_requirements)

  if contract_result == Failed:
    return VerificationOutcome::Failed(contract_result.reason)

  // PASS 2: Prediction comparison (LOADABLE, SOFT SIGNAL)
  // Compare actual outcome to predicted outcome
  if let Some(prediction) = intent.expected_outcome:
    prediction_match = compare_prediction(prediction, result.post_action_state)

    match prediction_match:
      Match(score > 0.8) →
        // Prediction was accurate — strong confidence in success
        // Learning signal: boost this intent type's prediction accuracy
        record_prediction_accuracy(intent, PredictionAccurate)
        return VerificationOutcome::Verified(confidence: High)

      PartialMatch(score 0.4-0.8) →
        // Prediction partially matched — something unexpected happened
        // Don't fail, but flag the surprise for learning
        record_prediction_accuracy(intent, PredictionPartial, delta: mismatch_details)
        return VerificationOutcome::Verified(confidence: Medium, surprise: mismatch_details)

      Mismatch(score < 0.4) →
        // Prediction was wrong — the world looks different than expected
        // Contract passed but prediction failed. This is a surprise.
        // Escalate to evidence-gathering sub-loop
        record_prediction_accuracy(intent, PredictionMismatch, delta: mismatch_details)
        return VerificationOutcome::Inconclusive(reason: PredictionMismatch)

  // No prediction generated — fall back to contract-only result
  return VerificationOutcome::from_contract(contract_result)

The Two-Pass Architecture

Pass 1: Artifact Contract (Kernel)

Purpose: Hard safety gate. Does the evidence meet the minimum requirements for "done"?

Examples: Confirmation dialog appeared. File exists at expected path. HTTP 200 received. Element with expected text is present.

Properties: Deterministic. Static. Same contract for every "tap button" action regardless of context. Cannot be bypassed.

Failure mode: If Pass 1 fails, the action failed. Period. Prediction is not consulted.

Pass 2: Prediction Comparison (Loadable)

Purpose: Intelligent verification. Does the world look like the agent expected? Catches subtle failures that pass contracts but indicate something is wrong.

Examples: Expected to see the home screen after pressing back, but see a dialog instead. Expected the message to appear in the conversation, but the conversation is empty. Expected the search to return restaurant results, but got an error page.

Properties: Context-dependent. Different prediction for every action based on the agent's understanding of the current state. Soft signal — informs learning, doesn't hard-gate.

Failure mode: If Pass 2 mismatches, it doesn't mean the action failed — it means something unexpected happened. Could be: (a) the agent's model of the world was wrong, (b) a subtle failure the contract didn't catch, or (c) a benign UI variation. The mismatch triggers investigation, not automatic failure.

Why two passes instead of one: Safety and intelligence have different requirements. The artifact contract is a hard floor — it prevents the agent from falsely claiming success. The prediction comparison is a soft ceiling — it tells the agent "yes, you succeeded, but was it what you expected?" Combining them gives both reliability (never miss a real failure) and intelligence (catch subtle failures and learn from them).

Prediction Formats

Predictions can be expressed in multiple formats depending on the action type and the available compute.

FormatWhen to useComparison methodCost
Structured diff Screen state predictions where elements are identifiable (element IDs, text content, hierarchy position) Deterministic comparison: expected elements present? Expected text matches? Expected hierarchy position correct? Low (no LLM needed for comparison)
Natural language Complex predictions where the expected state is hard to express structurally ("the conversation should show my sent message with a delivery indicator") LLM-based comparison: give the model the prediction + actual state, ask if they match Medium (LLM call for comparison)
Embedding similarity Visual predictions where overall screen appearance matters more than specific elements Embedding distance between predicted screen state and actual screen state. Threshold-based match. Low-Medium (embedding generation, vector comparison)

ExpectedOutcome {

        content_contains: "Happy birthday Mom!",

      },
      ElementPrediction {

      },
    ],
    // The compose box should be empty
    expected_absent: vec!["draft text in compose field"],
  }),
  confidence: 0.85,
  format: PredictionFormat::Structured,
}

Learning Signal: Prediction Accuracy as Intelligence Metric

Prediction accuracy over time is one of the most powerful metrics for agent intelligence. An agent that can predict the outcome of its actions accurately understands its environment. One that can't is guessing.

What Prediction Accuracy Tells Us

PatternMeaningResponse
High accuracy across the board Agent has a good model of the environment and apps it operates in Increase prediction confidence thresholds. Start using predictions to skip unnecessary verification steps.
Low accuracy for specific app Agent doesn't understand this app's behavior well Weight this app's actions toward more conservative verification (full screen reads). Feed app-specific prediction errors into dreaming for pattern extraction.
Accuracy was high, now dropping Something changed — app updated, workflow changed, or environment shifted Flag as anomaly. Reset prediction confidence for this action type. Gather more evidence for upcoming actions.
High accuracy for user-specific patterns Agent is learning the user's specific environment (not just generic app behavior) This is the sign of genuine personalization. The identity model is working.

Cost Analysis

Generating predictions adds cost. Here's the analysis:

ConcernImpactMitigation
Token cost for prediction generation ~50-200 extra tokens per intent during Reason step Prediction generation is part of the Reason prompt — marginal cost when the LLM is already reasoning about what to do. Structured predictions cost fewer tokens than NL.
Token cost for NL comparison ~100-300 tokens per NL comparison during Verify step Only used for NL-format predictions. Structured and embedding comparisons are free (local compute). Default to structured predictions; use NL only when structured is insufficient.
Latency for verification Structured: ~0ms. NL: ~500ms. Embedding: ~50ms. Structured comparison is instant. NL comparison can run async while the loop continues to the next intent. Only block on mismatch.
Overall budget impact ~5-15% increase in per-action token cost Offset by: (a) fewer retries (mismatches caught early), (b) faster verification (prediction match = instant verify), (c) better strategies over time (prediction accuracy learning).

Kernel vs Loadable Split

ComponentLayerRationale
Artifact contract checking (Pass 1) Kernel Safety gate. Must always run. Cannot be bypassed or weakened.
Prediction generation Loadable Intelligence. Different strategies for different action types. Should evolve via A/B slots.
Prediction comparison Loadable Intelligence. Comparison method should be configurable (structured/NL/embedding).
Prediction accuracy tracking Kernel Data collection. Kernel records every prediction result for the learning pipeline. Cannot be disabled.
Mismatch escalation policy Kernel Mismatch → Inconclusive is kernel-decided. What happens next (evidence gathering sub-loop) is kernel-orchestrated.
Prediction format selection Loadable Strategy for which format to use per action type. Should adapt based on accuracy data.

Invariants

🔒 Artifact contracts are never weakened by predictions. Pass 1 (artifact contract) is the hard gate. A successful prediction cannot override a failed contract. Predictions only add intelligence on top of safety, never replace it.
🔒 Prediction accuracy is always recorded. Even if predictions are disabled for a category, the comparison data (when available) is logged for the learning pipeline. This data is too valuable to discard.
🔒 Predictions are optional per-intent. The system works without predictions (falls back to contract-only verification). A failed or missing prediction never blocks the loop.
Design principle: Expectation-verification transforms the agent from "did it work?" to "did it work the way I expected?" That's a fundamentally different question — and the gap between prediction and reality is where learning happens. An agent that can predict accurately understands its environment. An agent that tracks its prediction accuracy knows how well it understands. An agent that feeds both into its learning cycle improves at an accelerating rate.
