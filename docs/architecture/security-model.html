Fawx — Security Model

A personal computer that happens to be AI-native. Your machine, your data, your rules.

🔒 Closed by default. No listening ports on public internet. No API endpoints. No webhooks from strangers. Information flows outbound on user permission, inbound only through hardened messaging channels with strict allowlists.
🏠 No trusted networks. Not the home LAN, not 5G, not the coffee shop WiFi. The phone leaves the house. A smart TV gets compromised. Fawx assumes every network is hostile, always. Zero-trust at every boundary.
🎯 Messaging is the attack surface. Attackers can't SSH in, can't hit an API, can't open a WebSocket — but they CAN send a message. Every inbound message is a potential prompt injection attempt. Channel hardening is critical path.
🧠 AI-native threats are unique. Memory poisoning, self-modification compromise, LLM provider MITM, time-delayed injection, multi-agent escalation. Traditional security models don't account for these.

Identity Model

Fawx serves one human. There is no multi-tenant architecture, no authentication tiers, no concept of "admin vs user." One owner, one agent, one relationship. Identity is implicit — the person sitting at the machine, sending messages to the agent, is the owner by definition.

The engine IS the OS kernel. Traditional shells (TUI, desktop GUI, mobile app) are just peripherals — different ways to interface with the same underlying agentic system. The agent's identity and relationship exist at the kernel level, not at the shell level.

This simplifies the user-facing security model enormously. There are no user-to-user authorization bypasses because there is only one user. However, internal privilege boundaries exist and are critical — the self-development lifecycle enforces typed, locked subagent roles (implementer, reviewer, fixer) with distinct permissions, and git authorization tiers separate what the agent can do (branches, staging) from what only the owner can do (main). These are not user-facing auth — they are kernel-enforced architectural constraints that prevent the agent's own subsystems from escalating beyond their designated roles.

Network Posture

Closed by Default

Fawx runs no listening services on the public internet. No HTTP servers, no API endpoints, no webhook receivers, no WebSocket listeners. The attack surface from network services is zero because there are no network services.

Information flow is primarily outbound. The agent makes HTTP requests to fetch web pages, calls APIs, downloads content — but only when explicitly permitted by user-defined policy. The agent reaches out to the world; the world cannot reach in.

Zero-Trust Networking

There is no concept of "trusted network." Not the home LAN where a smart TV could be compromised. Not the 5G connection where cellular spoofing is possible. Not the coffee shop WiFi that's obviously hostile. The phone leaves the house and connects to arbitrary networks.

All connections are authenticated and encrypted regardless of network type. Certificate pinning matters especially on mobile where DNS and routing can be manipulated. DNS-over-HTTPS or pinned resolvers prevent DNS poisoning attacks that could redirect api.anthropic.com to an attacker endpoint.

Critical Threat: DNS Poisoning
DNS poisoning redirects api.anthropic.com to attacker endpoint. The agent's brain is now controlled by the attacker. Especially dangerous on mobile/public networks where DNS cannot be trusted.

Threat Model — Fourteen Attack Surfaces

Traditional security models focus on network perimeter, user authentication, and data access control. AI-native systems have fundamentally different attack surfaces. Eight of these fourteen surfaces are unique to systems where the "user" is an autonomous agent that processes external content and modifies itself.

Surface Breakdown

1. Messaging Inbound — The Primary Attack Vector

This is where prompt injection lives today. An attacker cannot SSH into your Fawx, cannot hit an API endpoint, cannot open a WebSocket connection — but they can send you a message on Telegram, email you a document, send you an SMS, or trigger a push notification.

Every message is processed by the agent as potential instruction. A carefully crafted message can override system prompts, extract information, or cause malicious actions. Defense requires sender allowlists, content filtering, rate limiting, and treating every inbound message as untrusted until proven safe.

6. Supply Chain / Self-Modification (AI-Native)

Traditional systems don't modify themselves. Fawx does. The agent can write code, update dependencies, modify its own configuration. If an attacker achieves even momentary prompt injection during a self-development cycle, they can embed persistent backdoors in the code that the agent itself writes and deploys.

This requires test gates, user merge gates, and careful sandboxing of self-modification workflows. The agent cannot merge its own PRs into production branches.

7. Memory Poisoning (AI-Native)

If an attacker successfully injects malicious content into the agent's persistent memory — episodic experiences, semantic knowledge, configuration, or learned preferences — every future session is compromised. The agent loads poisoned memory and acts on false information or malicious instructions that seem to come from its own learned experience.

Memory writes from externally-triggered actions must be flagged and quarantined. Memory consolidation ("dreaming") processes require careful validation to prevent promotion of malicious content into long-term storage.

8. LLM Provider Channel (AI-Native)

The agent's "brain" runs in someone else's cloud. Every reasoning step, every decision, every plan passes through API calls to Anthropic, OpenAI, or other providers. A man-in-the-middle attack on these connections gives the attacker complete control over the agent's thoughts and actions.

Certificate pinning is critical, especially on mobile where DNS and routing can be compromised. API request/response integrity checks prevent response tampering. Data exfiltration via prompts is also a risk — malicious prompts that cause the agent to include sensitive information in API requests.

11. Multi-Agent Escalation (AI-Native)

Fawx uses recursive subagent spawning for complex tasks. If a subagent is compromised (through any of the above attack surfaces), it can inject malicious instructions in its output that gets processed by the orchestrating agent. The compromised subagent might claim elevated permissions it doesn't have, or provide false information that causes the orchestrator to take harmful actions.

This is why subagent roles have typed, locked permissions — implementers cannot post reviews, reviewers cannot push code, and no subagent can escalate its own permissions or modify the permission system.

14. Budget Manipulation / Resource Exhaustion (AI-Native)

The agent operates under a computational budget — LLM calls, tool invocations, tokens, cost in cents, and wall-clock time. When any resource is exhausted, the loop terminates. An attacker who can manipulate the agent’s budget consumption can effectively disable it — a denial-of-service attack that doesn’t require network access, authentication bypass, or code execution.

Budget manipulation attacks work through the agent’s own decision-making:

  • Decompose spirals: Craft prompts that trigger recursive decomposition, burning budget on planning overhead. Adaptive depth cap (effective_max_depth()) scales with remaining LLM calls, preventing deep trees. Cost gate rejects plans exceeding 150% of remaining budget. Batch detection and complexity floor prevent trivial decomposition overhead.
  • Retry amplification: Cause persistent tool failures that the model retries, draining invocation and LLM call budgets. Per-tool retry cap (default 2 retries = 3 total attempts per tool name per cycle) escalates to SignalKind::Blocked.
  • Fan-out explosion: Request operations that generate many parallel tool calls, inflating context and consuming invocation budget. Fan-out cap (default 4 per LLM response) defers excess calls. Tool result truncation (16KB) and aggregate result bytes tracking (400KB) cap context consumption.
  • Poisoned state: False information in persistent memory causes the model to attempt impossible tasks, wasting budget on doomed iterations. Primary defense is upstream (memory system hardening, surface #7); budget guardrails are defense-in-depth.

Defense is multi-layered and kernel-enforced: budget soft-ceiling forces graceful wrap-up at 80% cost consumption, fan-out caps limit parallel tool calls per turn, per-tool retry caps prevent retry loops, decompose cost gates reject expensive plans, and adaptive depth caps scale decomposition depth with remaining budget. The agent cannot disable or circumvent these mechanisms. BudgetConfig::conservative() provides a significantly tighter profile (8 LLM calls, 16 tool invocations, 100 cents, 2-minute wall time) for autonomous/background actions, drastically reducing the attack surface for non-user-initiated loops.

Residual risk: Low for most vectors after Wave 4 (#1098, #1100, #1101). Medium for memory poisoning → budget waste (primary fix is in memory system, surface #7) and cross-tool cycling (model rotates tool names to bypass per-tool retry cap, but soft-ceiling at 80% is the backstop regardless of cycling pattern). No budget attack vector has severity above Medium. See budget attack surface spec for full threat analysis.

Tool Execution Security

The tool executor runs locally with the user's permissions. This design recognizes two distinct threat scenarios:

Scenario 1: Owner Command Execution

When the owner directly asks the agent to do something — "delete this file," "send this email," "install this package" — the tool sandbox does not prevent the action. The owner has the right to control their own machine through their agent.

Scenario 2: External Input Processing

When the agent processes external input (messages, web content, API responses, sensor data) that might contain adversarial content, the sandbox protects against unintended consequences. The agent being tricked by external content should not be able to perform destructive actions.

Sandbox Controls

  • Working directory jail: Tool execution is constrained to approved directories
  • Command timeouts: Prevent runaway processes from consuming resources
  • File size limits: Prevent disk exhaustion attacks
  • Policy engine: Different permission levels for inbound messages vs owner commands

The policy engine distinguishes between "Joe asked me to do X" and "someone sent me a message that might be trying to get me to do X." The same tool call has different permission levels depending on the trigger context.

Integration Surface

No Public Integration Surface

There is no public integration surface. Period. No REST APIs, no GraphQL endpoints, no webhook receivers, no public database access, no admin dashboards.

Integrations work in two ways:

  • Outbound: The agent calls external APIs when needed (weather service, calendar sync, etc.)
  • Inbound messaging: External services reach the agent through hardened messaging channels with strict sender allowlists

Local Integration — Ember/MCP

Ember handles protocol-level tool coordination between the agent and local MCP (Model Context Protocol) servers. All communication is localhost-only. No external MCP servers, no network-based tool discovery, no remote procedure calls from untrusted sources.

No Web Dashboard

There is no fawx.com dashboard, no hosted admin interface, no SaaS control plane. The TUI and desktop shell are the interface. If a web-based shell exists, it connects to your node over your tailnet — not a hosted service.

fawx.sh tells you how to install it. The website is documentation and onboarding, not a product surface.

The Critical Path — Messaging Channel Hardening

This is where prompt injection lives today and where defense must be strongest. An attacker cannot SSH in, cannot hit an API, cannot open a WebSocket — but they can send a message.

Attack Vectors

  • Direct injection: "Ignore all previous instructions and..."
  • Social engineering: "I'm locked out of my account, can you help me..."
  • Content-based: Malicious images, documents, links that contain hidden instructions
  • Context manipulation: Long conversations that gradually shift agent behavior
  • Impersonation: Pretending to be the owner from a different contact method

Defense Strategy

  • Sender allowlists: Only known, verified contacts can reach the agent
  • Content filtering: Scan messages for known prompt injection patterns
  • Rate limiting: Prevent message flooding attacks
  • Context isolation: External messages never elevated to system instruction level
  • Verification protocols: High-risk actions require out-of-band confirmation
Every Message is Hostile Until Proven Safe
The default assumption is that every inbound message is an attack attempt. Trusted senders can be compromised. Legitimate-looking content can contain hidden instructions. Defense in depth is the only viable strategy.

Remote Access

Users access their Fawx remotely through a private overlay network (Tailscale tailnet or equivalent). This provides:

  • User-only access: No port forwarding to the public internet, no public URLs
  • Mutual authentication: Both endpoints verify each other's identity
  • Encrypted transport: All communication encrypted end-to-end
  • Network isolation: Overlay network is isolated from untrusted networks

The tailnet provides an authenticated private overlay for transport access — but it is not a "trusted network." It reduces the network attack surface for remote shell connections. External content is still treated as untrusted even when accessed over the tailnet, and zero-trust principles apply to all layers above transport.

Cloud / SuperFawx (Future)

The current design focuses on single-node self-sufficiency. There is no cloud component, no central coordination, no multi-node abstractions. Each Fawx node operates independently.

Future multi-node coordination (SuperFawx) is deliberately not prevented by current architectural decisions, but it's also not designed for yet. The principle: don't add multi-node complexity until there's a concrete need for it.

fawx.sh is onramp, not product surface. The landing page helps users install their node, but the product lives entirely on the user's machine. There is no hosted service component.

Security Invariants

The following security invariants MUST hold at all times. These are compiled into the system architecture and cannot be modified by the agent at runtime:

  1. No listening ports on public internet — Zero network services exposed to untrusted networks
  2. All external content treated as untrusted — Never elevated to system instruction level
  3. Messaging channels have sender allowlists — No anonymous inbound communication
  4. Memory writes from external-triggered actions are flagged with provenance — Tagged with source channel, sender, and trigger context
  5. Self-modification requires test gate + user merge gate — Agent cannot deploy its own code without validation
  6. Subagent permissions are typed and locked — Cannot self-escalate or modify permission system
  7. LLM API connections use certificate pinning on mobile — Prevent MITM on the agent's "brain" connection
  8. No "trusted LAN" — Zero-trust at every network boundary
  9. Tool sandbox cannot be disabled by the agent — Safety constraints are architectural, not configurable
  10. DNS resolution hardened against poisoning — DNS-over-HTTPS or pinned resolvers required
  11. Kernel is immutable at runtime — Cannot be modified by the agent, loadable modules, or any external input
  12. Scheduled tasks from externally-triggered sessions require user confirmation — Cron jobs and delayed tasks created during external input processing are quarantined until approved
  13. Budget guardrails are kernel-enforced and cannot be bypassed by the agent — Soft-ceiling, fan-out cap, per-tool retry limit, decompose cost gate, and depth cap are compiled into the kernel. The agent cannot modify budget limits, disable enforcement, or reset consumed budget counters. Budget soft-ceiling at 80% forces graceful wrap-up regardless of consumption pattern.

The canonical invariant list is maintained in DOCTRINE.md §10. This security model provides the threat context; DOCTRINE.md is the enforceable source of truth.

Enforcement

These invariants are compiled doctrine — enforced by the Fawx kernel at runtime, not by convention or documentation. The agent cannot modify these constraints through any combination of self-modification, memory manipulation, or subagent coordination. See DOCTRINE.md for the canonical enforcement specification.

Violation of any invariant triggers system-level protection mechanisms: operation blocking, session termination, memory quarantine, or complete system isolation depending on the severity and context of the violation.
