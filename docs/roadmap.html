Phase 5: Distributed Fleet Post-Launch
Wire fx-fleet for real multi-node orchestration. Deferred from Phase 0 — requires fleet client agent.
0/5 done›
Fleet client agent
Lightweight daemon that runs on remote nodes (Mac Mini, MacBook, future VPS). Heartbeats to coordinator, receives tasks, reports results. Must be a separate binary (fawx-node), not the full engine.
›
NodeRegistry wiring
Wire NodeRegistry from config (auto-register known nodes). Heartbeat processing. Stale detection + status reporting. /fleet TUI command for visibility.
›
TaskRouter + Orchestrator wiring
Capability-based task routing to fleet nodes. Task lifecycle (submit → dispatch → running → complete/failed). Retry on node failure. Complements node_run (SSH) with structured task dispatch.
›
Binary rollback via ripcord
Extend fawx-ripcord to snapshot + restore the fawx binary itself. Versioned binary storage (keep last 3, symlink swap). Triggered by canary or manual. Deferred from Phase 0 — binary bugs are CI/release failures, not agent-caused regressions.
›
Fleet security hardening
mTLS between coordinator and nodes. Task signing (coordinator signs, node verifies). Capability allowlists per node. Audit log for cross-node operations.
