PHASE 1 Loop Composition (Active)

Replace implicit loop control flow with typed contracts. The loop asks components what to do instead of guessing from side effects.

#1644 Replace stringly-typed ActionStep with Action trait foundation laid
PR #1648 introduced typed action vocabulary:
  • ActionNextStep::Continue(ActionContinuation) / Finish(ActionTerminal) — act step owns its disposition
  • TurnCommitment (ProceedUnderConstraints, NeedsDirection) — typed state across iterations
  • LoopResult::Incomplete — explicit terminal for non-completion
  • Loop no longer infers continuation from tool presence
~75% — ActionNextStep drives the main run_cycle. But 6+ code paths in act_with_tools still branch on tool_results.is_empty() for budget accounting, turn counting, and observation tracking. Those are secondary to the primary loop signal but still represent old-style implicit dispatch.
#1641 Decompose loop_engine.rs into composable phase traits foundation laid
PR #1648 extracted 4 modules with distinct responsibilities:
  • bounded_local.rs — phase machine (Discovery → Mutation → Recovery → Verification → Terminal)
  • continuation.rs — turn commitment rendering, tool scope resolution
  • direct_utility.rs — manifest-driven direct utility routing
  • progress.rs — progress event emission, turn state tracking
Main file grew from ~19K to 24K lines with new orchestration logic. The 4 submodules are impl LoopEngine blocks in separate files, not trait-based phases. LoopEngine struct has 67 fields. No traits defined in any submodule. Composition improved (responsibilities are separated), but the issue asks for "composable phase traits" and no traits exist yet.
Depends on: #1644 (action types used by phase modules)
~15% — file-level organization done. Trait-based composition, struct decomposition, and size reduction all pending. Main file is larger than before.
#1614 Bug: double response when model emits text + tool call in same turn foundation laid
ActionNextStep provides the typed signal. However, in the inner act_with_tools loop, when a response has both text and tool calls, the code checks !response.tool_calls.is_empty() and continues — but the text from that mixed response is not explicitly preserved. The text ends up in continuation_messages only if the LLM provider includes it in the response content blocks. The fix needs to ensure mixed responses are handled at the ActionNextStep level, not the inner tool round level.
Depends on: #1644 (ActionNextStep)
~20% — typed signal exists, but the actual mixed-response handling in act_with_tools is unchanged
PHASE 2 Tool Trait Decomposition

Break the FawxToolExecutor monolith into individual Tool trait objects. Each tool self-describes its schema, cacheability, classification, and journal hints.

#1639 Tool trait: replace FawxToolExecutor monolith with trait-based dispatch foundation laid highest impact
PR #1648 added 3 new methods to ToolExecutor trait:
  • classify_call() — observation vs mutation, with default delegating to cacheability()
  • route_sub_goal_call() — safe direct routing for zero-arg tools
  • cacheability() — pre-existing, unchanged
These are methods on the monolithic executor, not on individual Tool trait objects. FawxToolExecutor is still 5,938 lines with a 27-arm match on call.name.as_str() at line 305. No Tool trait exists. The Skill trait for WASM skills is the reference pattern but hasn't been generalized to built-in tools. Manifest-driven x-fawx-direct-utility metadata proves the skill-declares-behavior pattern works end-to-end (manifest → schema → kernel reads it).
Unlocks: #1638 (ripcord registries), #1515 (journal coverage), #1646 (visible skill tools), #1171 (WASM tool definitions)
~10% — 3 trait methods added to monolithic executor. No Tool trait, no decomposition, 27-arm match intact.
Enhancement scope: Structured memory system (DeerFlow-style) attaches here. Memory becomes a trait-based tool with typed add/replace/remove actions.
#1638 Audit: replace static dispatch with trait-based polymorphism foundation laid
Parent audit issue. classify_call() on the executor trait is one step toward executor-declared behavior, and manifest metadata proves the pattern for skills. But all critical instances are untouched:
  • extract_journal_action in ripcord — static string registry
  • tool_to_action_category in ripcord — parallel static registry
  • fawx_tool_definitions — manual Vec assembly
  • cacheability_for — 20+ arm match on tool name strings
  • Scattered ToolDefinition builders in git_skill.rs, session_tools.rs
These all depend on #1639 (Tool trait). Until individual tools are trait objects, the registries have nothing to delegate to.
Depends on: #1639 (tool trait provides the polymorphic contract)
~5% — audit complete, one trait method added, all critical instances untouched

PHASE 4 Skill Ecosystem

Leverages the trait-based architecture from Phases 1-2 to build a composable skill system.

#1646 Direct utility routing and visible tool contracts for loadable skills foundation laid
PR #1648 built the full metadata pipeline for 1 skill (weather). Three phases remain:
  • Phase 1: Generalize profiles. Replace hardcoded DirectUtilityProfile enum (Weather, CurrentTime) with data-driven struct built from manifest metadata. Eliminate 5 match-arm functions. Add is_structured_tool_schema() guard against legacy wrappers.
  • Phase 2: Migrate legacy skills. 3 of 8 skills (calculator, canvas, github) still use legacy {"input": "string"} wrapper. Add [[tools]] sections to their manifests. Update encode_runtime_input() for structured arg routing.
  • Phase 3: Safety gate. Negative tests proving legacy-wrapped tools cannot activate direct utility routing.
Depends on: #1639 (tool trait for skill-as-tool contract), #1171 (WASM tool definitions)
~30% — metadata pipeline end-to-end for weather. Profile enum hardcoded, 3 legacy skills remain, no structured arg routing for manifest tools.
#1647 Revisioned skill activation lifecycle not started
No revision tracking, source awareness, or activation management exists. Three phases:
