PHASE 1 Loop Composition (Active)

Replace implicit loop control flow with typed contracts. The loop asks components what to do instead of guessing from side effects.

#1644 Replace stringly-typed ActionStep with Action trait substantially complete
PR #1648 introduced typed action vocabulary:
  • ActionNextStep::Continue(ActionContinuation) / Finish(ActionTerminal) — act step owns its disposition
  • TurnCommitment (ProceedUnderConstraints, NeedsDirection) — typed state across iterations
  • LoopResult::Incomplete — explicit terminal for non-completion
  • Loop no longer infers continuation from tool presence
~85% — remaining: wire perception actions through the same typed path
#1641 Decompose loop_engine.rs into composable phase traits foundation laid
PR #1648 extracted 4 modules with distinct responsibilities:
  • bounded_local.rs — phase machine (Discovery → Mutation → Recovery → Verification → Terminal)
  • continuation.rs — turn commitment rendering, tool scope resolution
  • direct_utility.rs — manifest-driven direct utility routing
  • progress.rs — progress event emission, turn state tracking
Main file grew with new orchestration logic. Composition improved; size reduction pending.
Depends on: #1644 (action types used by phase modules)
~35% — modules extracted, but core orchestration still monolithic
#1614 Bug: double response when model emits text + tool call in same turn foundation laid
ActionNextStep provides the typed signal needed to fix this: the act step can now explicitly say "I have tool calls, continue" even when text is also present. The old heuristic (no tools = done) was the root cause.
Depends on: #1644 (ActionNextStep)
~40% — infrastructure ready, fix not yet wired
PHASE 2 Tool Trait Decomposition

Break the FawxToolExecutor monolith into individual Tool trait objects. Each tool self-describes its schema, cacheability, classification, and journal hints.

#1639 Tool trait: replace FawxToolExecutor monolith with trait-based dispatch foundation laid highest impact
PR #1648 added trait methods that define the future Tool contract:
  • classify_call() — executor classifies its own tool invocations (observation vs mutation)
  • route_sub_goal_call() — executor materializes direct tool calls from sub-goals
  • cacheability() — already existed, now part of a richer trait surface
  • x-fawx-direct-utility manifest metadata — skills declare routing to the kernel
The contract is defined. The decomposition (splitting 5,400 lines into individual Tool trait objects) is not started.
Unlocks: #1638 (ripcord registries), #1515 (journal coverage), #1646 (visible skill tools), #1171 (WASM tool definitions)
~25% — trait surface defined, monolith intact
Enhancement scope: Structured memory system (DeerFlow-style) attaches here. Memory becomes a trait-based tool with typed add/replace/remove actions.
#1638 Audit: replace static dispatch with trait-based polymorphism foundation laid
Parent audit issue. PR #1648 moved classification from static kernel knowledge to executor-declared behavior and routing from kernel heuristics to manifest-declared metadata. Remaining instances:
  • extract_journal_action in ripcord — static string registry
  • tool_to_action_category in ripcord — parallel static registry
  • fawx_tool_definitions — manual Vec assembly
  • Scattered ToolDefinition builders in git_skill.rs, session_tools.rs
These fall out naturally once #1639 lands.
Depends on: #1639 (tool trait provides the polymorphic contract)
~20% — audit done, pattern established, instances remain
