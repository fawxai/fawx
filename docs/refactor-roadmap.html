GOAL 1 Loadable Local Skill Substrate
Checkpoint verdict: real and promising — strongest of the four goals. Prioritize making it rock solid.
#1639 Tool trait: replace FawxToolExecutor monolith ACTIVE — critical path highest impact
Decompose FawxToolExecutor (5,938 lines, 27-arm match) into individual Tool trait objects with a registry-based executor. Each tool self-describes its name, schema, cacheability, classification, and journal hints. The executor becomes a registry that delegates to registered tools.
  • Tool trait: name, definition, execute, cacheability, classify_call, journal_action, action_category
  • ToolContext shared state struct
  • Tool implementations grouped by domain (filesystem, shell, web, session, git)
  • Registry-based dispatch replaces 27-arm string match
  • Deletes cacheability_for(), classify_call_impl(), fawx_tool_definitions() manual Vec
Unlocks: #1646, #1647, #1638, structured memory tool, WASM/built-in unification
~10% — 3 trait methods on monolithic executor. No Tool trait, no decomposition, 27-arm match intact.

  • Replace DirectUtilityProfile enum with data-driven struct from manifest metadata

  • Safety gate: negative tests proving legacy-wrapped tools cannot activate direct utility routing
Depends on: #1639

GOAL 2 Unified Authority Model
Checkpoint verdict: biggest architectural crack — no single authoritative answer to "what surface? what policy? who decides?" Multiple overlapping control planes can disagree.
NEW Unified capability/kernel boundary model spec complete
Spec: docs/specs/refactor/unified-authority-model.md (refined against source code)

Five overlapping control planes + duplicated tool-layer enforcement. 7 concrete conflict scenarios identified:
  • Kernel-blind read: permission gate allows, proposal gate blocks
  • Shell-read: permission gate reasons about shell, proposal gate inspects command text
  • git_checkpoint: unmapped in permission gate, mapped in proposal gate
  • Session approval keyed by tool name, not surface (approving one write blesses all writes)
  • CapabilityMode mutates path policy via config_bridge, not just presentation
  • Tool-layer self-modify checks duplicate kernel enforcement
  • Delete surface advertised but not fully modeled
5-phase implementation: parity matrix → canonical classification → shared resolver → attribution → wrapper collapse.
Benefits from: #1639 (tools self-describe their surface). Phases 0-2 can start after #1639 lands.
~15% — spec complete and source-validated. Implementation not started.
