GOAL 2 Unified Authority Model
Checkpoint verdict: biggest architectural crack — no single authoritative answer to "what surface? what policy? who decides?" Multiple overlapping control planes can disagree.
NEW Unified capability/kernel boundary model needs spec
Today, five systems can disagree about whether an action is allowed:
  • CapabilityMode (immediate denial vs prompt negotiation)
  • self_modify / kernel_modify flags
  • Proposal gating (sovereign vs non-sovereign surfaces)
  • Path policy (tool-level path restrictions)
  • Ripcord/tripwire (runtime behavioral constraints)
Need one canonical model:
  • Every action targets exactly one surface (loadable, sovereign, external)
  • Each surface has exactly one policy authority
  • The policy is communicated to both the loop and the user through a single mechanism
  • No two systems can contradict each other on the same action
This spec doesn't exist yet. It's the prerequisite for Goals 3 and 4.
Benefits from: #1639 (tools self-describe their surface), #1647 (skills declare their scope)
0% — no spec, no issue. Overlapping control planes remain.
