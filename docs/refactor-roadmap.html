#1614 Bug: double response when model emits text + tool call in same turn foundation laid
ActionNextStep provides the typed signal. However, in the inner act_with_tools loop, when a response has both text and tool calls, the code checks !response.tool_calls.is_empty() and continues — but the text from that mixed response is not explicitly preserved. The text ends up in continuation_messages only if the LLM provider includes it in the response content blocks. The fix needs to ensure mixed responses are handled at the ActionNextStep level, not the inner tool round level.
Depends on: #1644 (ActionNextStep)
~20% — typed signal exists, but the actual mixed-response handling in act_with_tools is unchanged
PHASE 2 Tool Trait Decomposition

Break the FawxToolExecutor monolith into individual Tool trait objects. Each tool self-describes its schema, cacheability, classification, and journal hints.

#1639 Tool trait: replace FawxToolExecutor monolith with trait-based dispatch foundation laid highest impact
PR #1648 added 3 new methods to ToolExecutor trait:
  • classify_call() — observation vs mutation, with default delegating to cacheability()
  • route_sub_goal_call() — safe direct routing for zero-arg tools
  • cacheability() — pre-existing, unchanged
These are methods on the monolithic executor, not on individual Tool trait objects. FawxToolExecutor is still 5,938 lines with a 27-arm match on call.name.as_str() at line 305. No Tool trait exists. The Skill trait for WASM skills is the reference pattern but hasn't been generalized to built-in tools. Manifest-driven x-fawx-direct-utility metadata proves the skill-declares-behavior pattern works end-to-end (manifest → schema → kernel reads it).
Unlocks: #1638 (ripcord registries), #1515 (journal coverage), #1646 (visible skill tools), #1171 (WASM tool definitions)
~10% — 3 trait methods added to monolithic executor. No Tool trait, no decomposition, 27-arm match intact.
Enhancement scope: Structured memory system (DeerFlow-style) attaches here. Memory becomes a trait-based tool with typed add/replace/remove actions.
#1638 Audit: replace static dispatch with trait-based polymorphism foundation laid
Parent audit issue. classify_call() on the executor trait is one step toward executor-declared behavior, and manifest metadata proves the pattern for skills. But all critical instances are untouched:
  • extract_journal_action in ripcord — static string registry
  • tool_to_action_category in ripcord — parallel static registry
  • fawx_tool_definitions — manual Vec assembly
  • cacheability_for — 20+ arm match on tool name strings
  • Scattered ToolDefinition builders in git_skill.rs, session_tools.rs
These all depend on #1639 (Tool trait). Until individual tools are trait objects, the registries have nothing to delegate to.
Depends on: #1639 (tool trait provides the polymorphic contract)
~5% — audit complete, one trait method added, all critical instances untouched
