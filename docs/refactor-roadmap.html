GOAL 2 Unified Authority Model
Checkpoint verdict: biggest architectural crack — no single authoritative answer to "what surface? what policy? who decides?" Multiple overlapping control planes can disagree.
NEW Unified capability/kernel boundary model spec complete
Spec: docs/specs/refactor/unified-authority-model.md (refined against source code)

Five overlapping control planes + duplicated tool-layer enforcement. 7 concrete conflict scenarios identified:
  • Kernel-blind read: permission gate allows, proposal gate blocks
  • Shell-read: permission gate reasons about shell, proposal gate inspects command text
  • git_checkpoint: unmapped in permission gate, mapped in proposal gate
  • Session approval keyed by tool name, not surface (approving one write blesses all writes)
  • CapabilityMode mutates path policy via config_bridge, not just presentation
  • Tool-layer self-modify checks duplicate kernel enforcement
  • Delete surface advertised but not fully modeled
5-phase implementation: parity matrix → canonical classification → shared resolver → attribution → wrapper collapse.
Benefits from: #1639 (tools self-describe their surface). Phases 0-2 can start after #1639 lands.
