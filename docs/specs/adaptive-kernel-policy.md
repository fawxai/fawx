# Adaptive Kernel Policy — Mutable Sandboxed Kernels

**Status:** Design spec  
**Depends on:** fx-forge, fx-telemetry, PermissionGateExecutor  
**Priority:** Post-ship, pre-forge-Phase-3  

---

## The Problem We Hit Today

Permission prompts on Cautious mode break the agentic loop. The model makes a tool call, waits for human approval, loses its flow, then spirals into retries. The kernel's safety policy is correct but its timing assumptions are wrong — it treats every tool call as equally risky and equally interruptible.

This isn't a model bug. It's a kernel policy that doesn't adapt to context.

## The Insight

An immutable kernel is essential for **runtime safety** — the agent can't weaken its own constraints mid-session. But kernel *policies* (not enforcement mechanisms) should evolve based on:

1. **Model behavior data** — which tools does this model use safely? (telemetry)
2. **User trust patterns** — what does this user always approve? (approval history)
3. **Session context** — an experiment needs different permissions than a chat
4. **Forge-trained models** — a model trained on successful tool use should earn broader trust

## Architecture: Two-Layer Kernel

### Layer 1: Immutable Enforcement (compiled, signed)

What CANNOT change at runtime, ever:
- Tier-3 path protections (kernel source blindness)
- Proposal gate existence (can't be removed)
- Permission prompt mechanism (can't be bypassed)
- Retry budget enforcement (can't be disabled)
- Chain integrity verification
- Sandbox boundaries

These are compiled `const` in the binary. No config, no env var, no admin override.

### Layer 2: Mutable Policy (sandboxed, auditable)

What CAN change between sessions or via controlled updates:
- Permission presets (Power / Cautious / Experimental / Custom)
- Per-tool-type auto-approval rules
- Retry budget values (2 retries? 5? depends on model reliability)
- Permission prompt timeout (30s? 120s? depends on UX context)
- Trust levels per tool category
- Session-scoped overrides ("allow all file reads this session")

Policy changes are:
- **Stored as data** (config/TOML), not compiled
- **Auditable** — every change logged with timestamp and reason
- **Reversible** — roll back to any previous policy
- **Sandboxed** — policy changes can't weaken Layer 1

## Adaptive Policy Engine

### Trust Scoring

Each tool type accumulates a trust score based on telemetry:

```rust
struct ToolTrustScore {
    tool_name: String,
    total_calls: u64,
    successful_calls: u64,
    denied_calls: u64,
    user_approved_calls: u64,
    user_denied_calls: u64,
    auto_approved_calls: u64,
    /// Computed: successful / total
    success_rate: f64,
    /// Computed: user_approved / (user_approved + user_denied)
    user_trust_rate: f64,
}
```

### Policy Recommendations

The telemetry system feeds trust scores into a policy recommender:

```
If tool X has:
  - 100+ calls
  - 99%+ success rate
  - 95%+ user approval rate
Then RECOMMEND: auto-approve tool X

If tool Y has:
  - user denied 3+ times this session
Then RECOMMEND: block tool Y for remainder of session
```

Recommendations are surfaced to the user: "Fawx has used `read_file` 200 times with 100% success. Auto-approve in future sessions?" The user decides. The kernel doesn't auto-escalate.

### Session-Scoped Overrides

For the immediate permission prompt friction:

```rust
enum SessionOverride {
    /// Auto-approve this specific tool for the rest of the session
    AllowToolForSession(String),
    /// Auto-approve all tools in this category
    AllowCategoryForSession(ToolCategory),
    /// Block this tool for the rest of the session
    BlockToolForSession(String),
}
```

The permission prompt UI offers:
- **Allow** — this call only
- **Allow for session** — auto-approve this tool type for the rest of the session
- **Deny** — this call only
- **Always deny** — block this tool for the session

Session overrides reset on restart. They never persist across sessions without explicit user action.

### Forge-Driven Policy Evolution

When forge trains a new model:
1. Model runs evaluation suite (fx-forge eval)
2. Eval includes tool-use scenarios with known-correct outcomes
3. Trust scores from eval inform default policy for that model
4. Policy ships as a "model policy profile" alongside the adapter

```toml
# Generated by forge eval, reviewed by human
[policy.profiles.fawx-lora-v1]
auto_approve = ["read_file", "search_text", "list_directory"]
ask_once_per_session = ["write_file", "edit_file", "run_command"]
always_ask = ["delete_file", "git_push"]
blocked = []
retry_budget = 3
permission_timeout_secs = 60
```

This is the bridge between forge (intelligence) and kernel (safety): forge produces the data that informs kernel policy, but a human reviews and approves the policy before it takes effect.

## Mutable Sandboxed Kernels

The radical extension: allow the agent to propose kernel policy changes that execute in a sandbox before being promoted.

### How it works:

1. **Agent proposes** a policy change: "I need `write_file` auto-approved for this task"
2. **Sandbox kernel** evaluates the proposal:
   - What's the blast radius? (just this session? permanent?)
   - Does it weaken Layer 1? (reject if so)
   - What's the trust score for this tool? (check telemetry)
3. **If safe**, the sandbox applies the policy change for a trial period
4. **Telemetry monitors** the trial: any anomalies → auto-revert
5. **After trial**, user is prompted: "Fawx used write_file 15 times during the trial with no issues. Make this permanent?"

### Sandbox boundaries:

- Sandbox kernel runs with the proposed policy IN ADDITION TO Layer 1
- Layer 1 constraints are NEVER relaxable, even in sandbox
- Sandbox has a time limit (max 1 hour) and call limit (max 100 tool calls)
- Sandbox telemetry is isolated — if things go wrong, we know exactly what the proposed policy caused
- Revert is instant and automatic on any safety signal

### What this enables:

- Agent learns its own optimal permission profile through experience
- Users aren't bombarded with permission prompts after the first session
- New forge-trained models can propose their own policies based on eval results
- The kernel stays safe because Layer 1 is immutable and the sandbox is bounded

## Implementation Phases

1. **Session-scoped overrides** — "Allow for session" in permission prompt UI. Immediate friction fix.
2. **Trust scoring** — telemetry tracks per-tool success/approval rates
3. **Policy recommendations** — surface recommendations to user based on trust scores
4. **Model policy profiles** — forge eval generates default policies per model
5. **Sandboxed policy proposals** — agent proposes, sandbox evaluates, user confirms

## The Principle

The kernel is not a wall — it's a membrane. Permeable in controlled ways, impermeable to unauthorized changes. Intelligence flows through it; safety constrains the flow. As the intelligence improves (forge), the membrane adapts (adaptive policy). But the membrane's existence and enforcement mechanisms are never in question.
